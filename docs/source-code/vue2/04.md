# 异步更新

当对数据进行更新操作时，比如 `obj.key = "new val"` 就会触发 `setter` 的拦截， 从而检测新值与旧值是否相等， 如果相等，则什么都不做；如果不相等，则更新值，然后后由 `dep` 通知 `watcher` 进行更新。 所以， `异步更新` 的入口点就是setter中最后调用的 `dep.notify()` 方法

### vue异步更新是如何实现的

### vue的 nextTick API是如何实现的

Vue.nextTick 或者 vm.$nextTick 就做了2间事情

- 将传递的函数 使用 try...catch 包裹， 然后放入到 callbacks数组
- 执行 timerFunc 函数， 在浏览器的异步任务队列放入一个刷新 callbacks 函数的数组

`src/core/util/next-tick.js`
```javascript
let callbacks = []

export function nextTick (cb?: Function, ctx?: Object) {
  let _resolve
  callbacks.push(() => {
    if (cb) {

      // 因为nextTick是提供给外部开发者可以直接调用，所以这里使用 try...catch 包装一层，保证程序的健壮性

      try {
        cb.call(ctx)
      } catch (e) {
        // vue内部检测到包装过的函数报错，就直接抛出异常
        handleError(e, ctx, 'nextTick')
      }
    } else if (_resolve) {
      _resolve(ctx)
    }
  })
  if (!pending) {
    pending = true
    timerFunc()
  }
  // $flow-disable-line
  if (!cb && typeof Promise !== 'undefined') {
    return new Promise(resolve => {
      _resolve = resolve
    })
  }
}
```

执行顺序
```javascript
nextTick --> timerFunc --> flushCallbacks
```

### watcher 为什么要从小到大排序

每一个 `watcher` 都有一个 `id` , `id` 从小到大，依次递增

见： `src/core/observer/scheduler.js` 中的 `flushSchedulerQueue` 方法

有3个目的

* 组件是从父组件更新到子组件的（因为： 父组件总是在子组件之前被创建）

> 父组件的watcher先被创建并执行， 再创建子组件的watcher;
>
> 假设： 子组件watcher先被创建并执行，再执行父组件， 则子组件watcher会被执行2次， 因为父组件更新后，会创建子组件，watcher会再次执行

* 让组件的用户watcher在其渲染(render watcher)监听器之前运行（因为`user watcher` (用户监听器)在`render watcher` (渲染监听器)之前被创建）
* 如果一个组件在其父组件的 watcher 运行期间被销毁，那么它的监听器可以被跳过。

### 为什么watcher在执行的时候没有缓存watcher的长度

`src/core/observer/scheduler.js`  `flushSchedulerQueue` 方法
```javascript
let queue = []
let index = 0

function flushSchedulerQueue() {
    queue.sort((a, b) => a.id - b.id)

    // 这里没有使用 , len = queue.length; index<len的方式缓存 队列的长度
    for (index = 0; index < queue.length; index++) {

    }
}
```

我们在刷新队列的时候， watcher 队列里面 `queue` 会持续不断的放进新的 watcher, 如果缓存了length, 则后面push进 队列 queue的 watcher没有办法得到更新（就是后面push进去的方法得不到执行）

所以，在 `for` 循环中， 队列长度总是采用`动态计算`的方式， 每次执行，都重新计算 queue 队列长度

#### performance.now 与 Date.now 区别

[performance.now - MDN](https://developer.mozilla.org/zh-CN/docs/Web/API/Performance/now)

* performance.now 精度高, 精度达到 1毫秒， 浮点数
* performance.now 不受系统时间影响
* performance.timing.navigationStart + performance.now() 约定于 = Date.now()
