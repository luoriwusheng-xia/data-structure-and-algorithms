# 01 - 平台架构设计

这一篇先不写代码，先回答一个更关键的问题：什么样的 Node.js 后端，才算一个可对外提供的 API 平台。

## 一、先区分三类系统

### 1. 普通业务后端

- 主要服务自家前端
- 调用方少
- 身份体系简单
- 版本演进压力小

### 2. 内部共享服务

- 服务多个内部系统
- 需要一定的 SLA
- 已经开始关注权限、限流、监控

### 3. 对外开放平台

- 调用方是外部企业、合作方、开发者
- 调用方式不可控
- 接口文档、鉴权、错误码、审计、版本治理必须标准化
- 平台要能承受长周期演进

本文讨论的是第 3 类。

## 二、平台的核心分层

推荐把平台拆成 5 层。

### 1. 接入层

负责：

- 域名与 HTTPS
- 负载均衡
- WAF / 基础风控
- IP 黑白名单
- 基础限流

这层通常由 Nginx、云网关、API Gateway、SLB 或 Ingress 承担。不要把所有接入治理逻辑都压在 Node 进程里。

### 2. 平台接入层

这一层是 Node.js 平台真正的入口，负责：

- API Key / Access Token 校验
- HMAC / RSA / Ed25519 签名校验
- 时间戳与 nonce 防重放
- 幂等键校验
- 参数校验
- requestId / traceId 注入
- 统一错误码

它不应该承载复杂业务决策，而是把“开放平台的通用治理”统一收口。

### 3. 领域服务层

负责具体业务：

- 订单查询
- 支付发起
- 用户信息同步
- 配额扣减
- 模型调用
- 文件处理

这一层才是“业务价值”所在。

### 4. 基础设施层

包括：

- MySQL / PostgreSQL：事务数据
- Redis：缓存、nonce、限流计数、幂等状态
- MQ：异步事件
- 对象存储：文件、回执、归档

### 5. 运维与治理层

包括：

- 结构化日志
- 审计日志
- 指标监控
- 告警
- 配置中心
- 密钥管理

很多团队只画前 4 层，漏掉第 5 层，结果平台上线后排障成本非常高。

## 三、推荐的逻辑架构

```text
Partner / Enterprise Client
        │
        ▼
  API Gateway / WAF
        │
        ▼
 Open Platform App (Node.js)
        │
        ├── Auth Module
        ├── Signature Verify Module
        ├── Idempotency Module
        ├── Rate Limit Module
        ├── Request Context Module
        │
        ▼
 Domain Services
        │
        ├── Order Service
        ├── Billing Service
        ├── Tenant Service
        ├── Webhook Service
        │
        ▼
 Infra
        ├── DB
        ├── Redis
        ├── MQ
        └── Object Storage
```

## 四、核心领域模型

如果平台要长期发展，至少要先建好下面几个模型。

### 1. Tenant（租户）

表示一个企业客户或组织。

常见字段：

- tenantId
- tenantName
- status
- contactEmail
- billingPlan
- environmentPolicy

### 2. App（应用）

一个租户下面通常会有多个接入应用：

- 测试环境应用
- 生产环境应用
- 不同业务线应用

常见字段：

- appId
- tenantId
- appName
- appEnv
- apiKey
- publicKey / secret
- status

### 3. Credential（凭证）

用于承载：

- API Key
- Client Secret
- 公私钥
- Token 策略

凭证应支持轮换，不要只存一套永久密钥。

### 4. Permission（权限）

决定：

- 哪个应用能访问哪些 API
- 能用哪些版本
- 配额是多少
- QPS 上限是多少

### 5. Audit Event（审计事件）

记录关键动作：

- 创建应用
- 旋转密钥
- 修改 IP 白名单
- 调整限流
- 手工重发 Webhook

## 五、接口平台与普通 REST 服务的关键差异

| 维度 | 普通 REST 服务 | 对外 API 平台 |
|------|----------------|----------------|
| 调用方 | 内部系统 | 外部企业 / 开发者 |
| 身份体系 | 登录态、服务间信任 | API Key、签名、令牌 |
| 错误治理 | 业务可协商 | 必须标准化 |
| 版本管理 | 可快速调整 | 必须兼容长期用户 |
| 日志审计 | 常被弱化 | 必须强约束 |
| 安全要求 | 中 | 高 |

## 六、接口设计的六个硬约束

### 1. 版本化

一旦开放出去，不要默认接口能随意改字段。

推荐做法：

- 路径版本：`/v1/orders/query`
- 或 Header 版本：`X-API-Version: 2026-07-01`

如果你的平台面向大量第三方，路径版本更容易治理和排障。

### 2. 幂等性

典型场景：

- 支付创建
- 退款申请
- 资源开通
- 回调重试

推荐通过：

- `Idempotency-Key`
- 请求摘要
- 状态表 / Redis

来保证多次请求只生效一次。

### 3. 限流

至少要支持：

- 全局限流
- 租户级限流
- 应用级限流
- 接口级限流

### 4. 安全隔离

至少考虑：

- 测试环境与生产环境隔离
- 不同租户数据隔离
- 管理接口与开放接口隔离

### 5. 异步回调

任何关键状态变化，都不要只依赖同步返回。应设计：

- Webhook 回调
- 重试
- 回调签名
- 死信处理

### 6. 可追踪

每个请求必须可还原：

- 谁发起
- 发到哪个接口
- 请求体是什么
- 验签结果
- 下游调用结果
- 最终响应

## 七、技术选型建议

站在 Node.js 26 的角度，我建议优先采用：

- 语言：JavaScript ESM 或 TypeScript ESM
- Web 框架：Express / Fastify / 原生 `node:http`
- 配置：环境变量 + 配置对象封装
- 数据库：PostgreSQL / MySQL
- 缓存：Redis
- 队列：RabbitMQ / Kafka / 云消息服务
- 日志：结构化 JSON 日志

如果平台强调高性能和插件化治理，Fastify 往往比 Express 更适合；如果你更重视团队认知成本和生态，Express 仍然足够。

## 八、推荐的部署形态

### 方案一：单体平台应用

适合：

- 平台刚起步
- 接口数量有限
- 团队人数不多

优点是实现快、维护简单。

### 方案二：平台接入层 + 领域服务拆分

适合：

- 接口数量增加
- 不同业务域复杂度明显上升
- 需要更清晰的边界

推荐方式：

- 平台接入层统一鉴权、签名、限流、上下文
- 领域服务按业务域拆分

### 方案三：网关化 + 事件驱动

适合：

- 高调用量
- 回调与异步事件很多
- 有独立平台团队

成本最高，但治理能力也最强。

## 九、架构决策建议

如果你正在做第一版，优先做下面这些，而不是一开始就拆十几个微服务：

1. 先把身份、签名、日志、审计、错误码统一起来
2. 先支持多应用、多环境
3. 先做幂等和回调重试
4. 先把敏感数据脱敏与密钥管理纳入设计
5. 先保证接口文档和平台规则稳定

这比盲目追求微服务形态更重要。

## 十、本篇结论

一个合格的对外 API 平台，本质上要同时解决四件事：

- 稳定接入
- 身份可信
- 权限可控
- 问题可追
