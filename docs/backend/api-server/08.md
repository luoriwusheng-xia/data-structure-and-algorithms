# 08 - 密钥轮换与凭证生命周期

如果一个开放平台上线后一直使用第一天生成的 secret，那么迟早会出问题。密钥不是静态配置，而是需要治理的生命周期对象。

## 一、为什么一定要支持轮换

典型场景：

- 凭证疑似泄漏
- 员工离职或外包交接
- 第三方内部系统迁移
- 平台算法升级
- 合规要求定期更新凭证

如果平台不支持平滑轮换，用户只能停机切换，这在企业对接场景里通常不可接受。

## 二、凭证生命周期建议

推荐状态机：

```text
issued -> active -> deprecated -> revoked -> archived
```

含义：

- `issued`：已生成，尚未正式启用
- `active`：当前可用
- `deprecated`：即将下线，但短期仍可接受
- `revoked`：吊销，不再接受
- `archived`：仅保留审计记录

## 三、推荐的轮换策略

### 方案一：双密钥并存

这是最常见、也最实用的方案。

流程：

1. 平台为 app 新增一组新凭证
2. 新旧两组凭证并存一段时间
3. 客户端完成切换
4. 平台停用旧凭证

优点：

- 对业务影响最小
- 适合大多数 API 平台

### 方案二：按 `keyId` 定位密钥

请求头里带：

```text
X-Key-Id: key_live_01J123XYZ
```

服务端根据 `keyId` 找到对应密钥进行验签。

这是推荐做法，因为：

- 轮换时不需要猜测该用哪把 key
- 更容易追踪和审计

## 四、推荐请求头

```text
X-App-Id: app_live_demo
X-Key-Id: key_live_01J123XYZ
X-Timestamp: 1720000000000
X-Nonce: nonce-001
X-Signature: abcdef...
```

服务端验签流程就变成：

1. 先识别 app
2. 再根据 `keyId` 找到具体凭证
3. 判断凭证状态
4. 用该 secret / public key 验签

## 五、轮换流程建议

### 1. 创建新凭证

新凭证生成后进入 `issued` 状态。

### 2. 客户端测试

调用方先在 sandbox 测试通过。

### 3. 双写兼容窗口

平台允许旧凭证和新凭证同时验签通过。

### 4. 切换窗口结束

旧凭证进入 `deprecated`，随后进入 `revoked`。

### 5. 审计归档

记录：

- 谁发起轮换
- 哪个 app
- 哪个旧 keyId
- 哪个新 keyId
- 切换开始和结束时间

## 六、服务端验证示例

```js
export async function resolveCredential({ appId, keyId }) {
  const credential = await credentialRepository.findActiveByAppAndKeyId(appId, keyId);

  if (!credential) {
    throw new Error('credential not found');
  }

  if (!['active', 'deprecated'].includes(credential.status)) {
    throw new Error('credential is not usable');
  }

  return credential;
}
```

如果你只允许 `active` 验签通过，那么就没有平滑轮换空间；因此通常 `deprecated` 在短窗口内也允许通过。

## 七、密钥存储建议

### 1. 不保存明文 secret

推荐保存：

- 加密后的密文
- 指纹
- 最后 4 位

### 2. 用独立的 KMS 或平台主密钥加密

不要只做“数据库字段 base64 编码”，那不叫加密。

### 3. 凭证读取要最小化

只有验签路径和少数平台管理动作需要读取解密后的 secret。

## 八、轮换时的兼容性处理

有些企业系统切换速度慢，平台需要显式说明：

- 轮换生效时间
- 旧凭证失效时间
- 回调验签使用哪组 key
- 文档更新窗口

如果平台支持 Webhook，回调方向也要支持凭证轮换，不然用户只切了“调用平台”的流量，却没切“平台回调用户”的流量。

## 九、吊销策略

吊销通常比轮换更紧急。

触发条件：

- 凭证泄漏
- 客户主动停用
- 风险事件
- 合规封禁

处理动作：

1. 立即改状态为 `revoked`
2. 平台拒绝后续请求
3. 触发高优先级告警
4. 通知租户管理员
5. 记录审计事件

## 十、日志与告警建议

至少记录这些事件：

- credential.issued
- credential.activated
- credential.deprecated
- credential.revoked
- credential.rotated
- credential.verify.failed

如果某个 `keyId` 的验签失败率突然暴涨，通常意味着：

- 对方切换不完整
- 时间戳漂移
- 密钥配置错误
- 有潜在攻击或泄漏

## 十一、本篇结论

密钥轮换不是可选项，而是开放平台的基础治理能力。平台设计上要从一开始就支持：

- 多凭证并存
- `keyId` 识别
- 状态机管理
- 审计与告警

没有这四项，平台后期凭证治理会非常被动。
