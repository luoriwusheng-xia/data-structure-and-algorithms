# 03 - 鉴权与授权设计

开放平台最容易做错的地方，不是业务逻辑，而是身份体系。

## 一、先区分 4 个概念

### 1. Authentication

调用方是谁。

### 2. Authorization

调用方是否有权限访问当前资源。

### 3. Signature Verification

请求是否被篡改，请求是否真的是凭证持有者发出的。

### 4. Replay Protection

这个请求是不是旧请求重放出来的。

很多项目把这四件事混成一件事，最后接口既不安全，也不好扩展。

## 二、开放平台常见鉴权模式

### 1. API Key

适合：

- 服务端到服务端调用
- 调用方数量可控
- 快速接入

问题：

- 只能识别身份，不能证明请求体未被篡改
- 泄漏风险高

### 2. Bearer Token / Access Token

适合：

- 有授权中心
- 需要 token 生命周期
- 希望细粒度授权

### 3. JWT

适合：

- 平台内部 token 流转
- 管理端或控制台登录态

不建议把“外部企业开放接口”的安全全部寄托在 JWT 上。外部开放接口更常见的是 API Key + 签名。

### 4. HMAC 签名

适合：

- 企业对企业接口
- 支付、计费、订单类强一致接口
- 需要防篡改、防重放

### 5. 非对称签名

适合：

- 安全要求更高
- 希望减少共享密钥暴露面
- 客户端持私钥，平台持公钥

## 三、推荐的企业开放平台鉴权模型

我建议采用两层设计：

### 第一层：应用身份识别

通过：

- `X-App-Id`
- `X-Key-Id`

识别是哪个租户、哪个应用在调用。

### 第二层：请求验签

通过：

- 时间戳
- nonce
- 请求方法
- 请求路径
- 请求体摘要
- HMAC / Ed25519 签名

证明请求未被篡改，且来自凭证持有者。

## 四、推荐请求头设计

```text
X-App-Id: app_live_xxx
X-Timestamp: 1720000000000
X-Nonce: 8f343df0a8334d2b
X-Signature: 4b19...
X-Signature-Method: HMAC-SHA256
Content-Type: application/json
```

如果已经发行 access token，也可以同时带：

```text
Authorization: Bearer <token>
```

## 五、签名原文如何设计

建议明确规定 canonical string，避免双方理解不一致。

例如：

```text
POST
/partner/v1/orders/query
1720000000000
8f343df0a8334d2b
sha256(requestBody)
```

然后用共享密钥生成 HMAC。

## 六、HMAC 签名示例

```js
import { createHash, createHmac, timingSafeEqual } from 'node:crypto';

function sha256Hex(content) {
  return createHash('sha256').update(content).digest('hex');
}

function buildCanonicalString({ method, path, timestamp, nonce, body }) {
  const bodyText = body ? JSON.stringify(body) : '';
  const bodyHash = sha256Hex(bodyText);

  return [method.toUpperCase(), path, timestamp, nonce, bodyHash].join('\n');
}

export function signRequest(input, secret) {
  const canonical = buildCanonicalString(input);
  return createHmac('sha256', secret).update(canonical).digest('hex');
}

export function verifySignature(input, secret, receivedSignature) {
  const expected = signRequest(input, secret);
  const left = Buffer.from(expected, 'utf8');
  const right = Buffer.from(receivedSignature, 'utf8');

  if (left.length !== right.length) {
    return false;
  }

  return timingSafeEqual(left, right);
}
```

这里有两个点不能省：

- 要对 body 做摘要，不要直接拼原始对象
- 比较签名时使用 `timingSafeEqual`

## 七、签名校验中间件示例

```js
import { AppError } from '../core/errors.js';
import { verifySignature } from '../core/signature.js';

export async function signatureMiddleware(req, res, next) {
  const appId = req.headers['x-app-id'];
  const timestamp = req.headers['x-timestamp'];
  const nonce = req.headers['x-nonce'];
  const signature = req.headers['x-signature'];

  if (!appId || !timestamp || !nonce || !signature) {
    return next(new AppError('missing signature headers', {
      code: 'SIGNATURE_HEADERS_MISSING',
      status: 400
    }));
  }

  const appCredential = await loadAppCredential(appId);
  if (!appCredential) {
    return next(new AppError('app not found', {
      code: 'APP_NOT_FOUND',
      status: 401
    }));
  }

  const valid = verifySignature(
    {
      method: req.method,
      path: req.path,
      timestamp,
      nonce,
      body: req.body
    },
    appCredential.secret,
    signature
  );

  if (!valid) {
    return next(new AppError('signature is invalid', {
      code: 'SIGNATURE_INVALID',
      status: 401
    }));
  }

  req.auth = {
    tenantId: appCredential.tenantId,
    appId: appCredential.appId
  };

  next();
}

async function loadAppCredential(appId) {
  return {
    appId,
    tenantId: 'tenant_001',
    secret: 'demo-secret'
  };
}
```

## 八、防重放设计

只验签还不够。攻击者可以复制合法请求，在短时间内重放。

推荐组合：

- `X-Timestamp`
- `X-Nonce`
- Redis 记录 nonce
- 固定时间窗口，例如 5 分钟

处理逻辑：

1. 检查请求时间与服务端时间差
2. 超出窗口直接拒绝
3. 检查 nonce 是否已使用
4. 已使用则判定为重放
5. 未使用则写入 Redis，并设置 TTL

示例：

```js
export async function assertReplaySafe({ redis, appId, nonce, timestamp, now, skewMs }) {
  const drift = Math.abs(now - Number(timestamp));
  if (drift > skewMs) {
    throw new Error('timestamp expired');
  }

  const key = `nonce:${appId}:${nonce}`;
  const used = await redis.get(key);
  if (used) {
    throw new Error('replayed request');
  }

  await redis.set(key, '1', { EX: Math.ceil(skewMs / 1000) });
}
```

## 九、授权模型

认证通过，不代表可以访问所有接口。

至少要判断：

- 当前 app 是否启用
- 当前 app 是否属于生产环境
- 是否允许调用目标 API
- 当前套餐是否支持该能力
- 当前配额是否足够

建议维护这样一套映射：

```text
tenant -> apps -> credentials -> permissions -> quotas
```

## 十、密钥管理建议

### 1. 支持多密钥并存

密钥轮换时，不要要求对方瞬时切换。应该支持：

- 当前生效密钥
- 上一个密钥短暂并存

### 2. 平台只保存密钥指纹或加密后的密钥

不要把明文 secret 随便写进日志、表单、管理后台页面。

### 3. 管理动作必须产生审计日志

比如：

- 创建凭证
- 启用 / 禁用凭证
- 旋转密钥
- 下载公钥

## 十一、常见误区

### 1. 只用 JWT 就开放给企业调用

这通常不够。JWT 更适合登录态和授权载体，不适合替代企业级请求签名。

### 2. 签名原文不固定

如果没有严格定义 canonical string，不同语言客户端会出现大量兼容问题。

### 3. 时间戳参与了签名，但不校验时间窗口

这等于没有真正防重放。

### 4. 通过字符串 `===` 比较签名

容易留下时序侧信道问题。

## 十二、本篇结论

开放平台鉴权最好拆成三步：

1. 识别应用身份
2. 校验请求签名
3. 判断是否有权调用

如果你做的是支付、订单、计费、AI 调用这类高价值接口，这三步不能混，也不能省。
