# 04 - 接口规范与安全治理

有了架构和鉴权，下一步是把平台规则定下来。接口平台的稳定性，很多时候不是来自某个框架，而是来自一致的协议和治理规则。

## 一、统一接口规范

建议平台从第一天开始就统一下面这些内容：

- 路径命名
- 版本策略
- 请求头规范
- 响应结构
- 错误码
- 幂等策略
- 分页格式
- 时间字段格式

## 二、路径与版本建议

推荐：

```text
/partner/v1/orders/query
/partner/v1/refunds/create
/partner/v1/webhooks/replay
```

建议：

- 用名词表达资源，用动词表达能力动作
- 对外平台优先采用显式版本路径
- 不要把内部领域模型直接暴露给外部

## 三、错误码体系

错误码不要只返回 HTTP 状态码。

推荐双层结构：

- HTTP 状态码：协议层
- 业务错误码：平台层

示例：

| HTTP | 平台错误码 | 含义 |
|------|------------|------|
| 400 | `INVALID_ARGUMENT` | 参数错误 |
| 401 | `SIGNATURE_INVALID` | 验签失败 |
| 401 | `TOKEN_EXPIRED` | token 过期 |
| 403 | `PERMISSION_DENIED` | 无权限 |
| 404 | `RESOURCE_NOT_FOUND` | 资源不存在 |
| 409 | `IDEMPOTENCY_CONFLICT` | 幂等冲突 |
| 429 | `RATE_LIMITED` | 被限流 |
| 500 | `INTERNAL_ERROR` | 平台内部错误 |

## 四、幂等设计

### 1. 哪些接口必须幂等

- 创建支付单
- 创建退款单
- 发货
- 开通资源
- 重试回调

### 2. 请求头规范

```text
Idempotency-Key: 6b3a1b1d-3a43-4b88-b99d-98fc60f4d7d7
```

### 3. 平台侧处理逻辑

1. 读取 `Idempotency-Key`
2. 生成业务范围 key，例如 `tenantId + path + key`
3. 查询 Redis / DB 是否已有记录
4. 已存在则返回历史结果
5. 不存在则创建处理中状态
6. 业务完成后保存最终响应摘要

示例：

```js
export async function idempotencyMiddleware(req, res, next) {
  const key = req.headers['idempotency-key'];
  if (!key) {
    return next();
  }

  const scopeKey = `idem:${req.auth?.tenantId}:${req.path}:${key}`;
  const existing = await req.redis.get(scopeKey);

  if (existing) {
    return res
      .status(200)
      .setHeader('x-idempotent-replayed', 'true')
      .json(JSON.parse(existing));
  }

  req.idempotencyScopeKey = scopeKey;
  next();
}
```

## 五、限流策略

至少分三层：

### 1. 平台基础限流

防止恶意流量和明显突刺。

### 2. 租户级限流

避免单个客户拖垮整个平台。

### 3. 接口级限流

保护高成本接口，例如：

- 生成图片
- AI 推理
- 结算

建议在响应头返回：

```text
X-RateLimit-Limit: 100
X-RateLimit-Remaining: 87
X-RateLimit-Reset: 1720001234
```

## 六、输入校验

开放平台必须假设调用方会传入各种异常参数。

至少校验：

- 字段存在性
- 类型
- 长度
- 枚举
- 金额精度
- 日期格式
- 业务边界

原则：

- 参数错误尽早失败
- 不要让脏数据进入业务层

## 七、敏感字段治理

这些字段默认要脱敏或禁止落日志：

- 手机号
- 身份证号
- 银行卡号
- token
- secret
- 签名原文中的密钥内容

可以保留：

- 哈希
- 截断值
- 掩码值

例如：

```js
function maskToken(value) {
  if (!value || value.length < 10) return '***';
  return `${value.slice(0, 4)}***${value.slice(-4)}`;
}
```

## 八、Webhook 安全治理

平台对外提供 Webhook 时，至少要保证：

- 回调请求带签名
- 支持重试
- 支持状态查询
- 支持手工重发
- 回调事件具备唯一 ID

推荐回调头：

```text
X-Webhook-Id: evt_001
X-Webhook-Timestamp: 1720000000000
X-Webhook-Signature: abcdef...
```

调用方校验签名后，再处理业务。

## 九、回调重试策略

建议明确：

- 哪些状态码触发重试
- 重试间隔
- 最大次数
- 死信处理

典型做法：

- 2xx：成功
- 4xx：通常不重试，除非业务特别定义
- 5xx / timeout：重试

重试时间可采用退避，例如：

```text
1m -> 5m -> 15m -> 1h -> 6h
```

## 十、环境隔离

开放平台至少要有：

- sandbox / test
- production

而且需要做到：

- 域名隔离
- 凭证隔离
- 数据隔离
- 回调地址隔离

不要让测试密钥误打到生产环境。

## 十一、接口文档建议

一个可用的对外 API 文档，至少应该包含：

- 接口用途
- 请求路径
- 请求方法
- 请求头
- 请求参数
- 响应示例
- 错误码
- 签名说明
- 幂等说明
- 示例代码

这类平台文档，本身就是产品的一部分。

## 十二、本篇结论

开放平台安全治理的重点不是“拦住一切”，而是让每一条规则都标准化、可执行、可排障。

把幂等、限流、签名、错误码、脱敏、回调规范收敛清楚，平台的可维护性会直接提升一个量级。
