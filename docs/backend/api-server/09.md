# 09 - Webhook 回调设计与验签

对外 API 平台通常不只接收请求，还要主动向第三方推送事件。这个能力就是 Webhook。

典型场景：

- 支付成功通知
- 退款状态变更
- 订单异步处理完成
- 配额变更
- AI 任务完成

如果回调设计得不好，平台会长期陷入“状态不同步、重复推送、签名争议、支持成本高”的问题。

## 一、Webhook 设计目标

一个成熟的 Webhook 机制至少要满足：

- 事件唯一可追踪
- 可签名验证
- 可重试
- 可查历史
- 可手工重放

## 二、推荐事件模型

建议事件体包含：

```json
{
  "eventId": "evt_20260701_0001",
  "eventType": "payment.succeeded",
  "occurredAt": "2026-07-01T08:00:00.000Z",
  "tenantId": "tenant_demo",
  "data": {
    "orderNo": "ORDER_10001",
    "amount": 19900,
    "currency": "CNY"
  }
}
```

关键点：

- `eventId` 全局唯一，用于幂等
- `eventType` 清晰表达语义
- `occurredAt` 表示业务发生时间，而不是投递时间

## 三、回调请求头建议

```text
Content-Type: application/json
X-Webhook-Id: evt_20260701_0001
X-Webhook-Timestamp: 1760000000000
X-Webhook-Signature: abcdef...
X-Webhook-Signature-Method: HMAC-SHA256
```

这套头部和你平台入站 API 的验签模型最好保持一致，方便调用方理解。

## 四、签名原文建议

推荐 canonical string：

```text
POST
/partner/webhook/payment
1760000000000
sha256(requestBody)
```

然后用 HMAC 生成签名。

## 五、平台侧签名示例

```js
import { createHash, createHmac } from 'node:crypto';

function sha256Hex(text) {
  return createHash('sha256').update(text).digest('hex');
}

export function signWebhook({ method, path, timestamp, bodyText }, secret) {
  const canonical = [
    method.toUpperCase(),
    path,
    timestamp,
    sha256Hex(bodyText)
  ].join('\n');

  return createHmac('sha256', secret).update(canonical).digest('hex');
}
```

## 六、投递流程建议

推荐把 Webhook 当成独立子系统处理，而不是在主业务流程里直接 `fetch` 一下就结束。

```text
业务事件产生
   -> 写入 webhook_event
   -> 推送任务入队
   -> worker 拉取任务
   -> HTTP 投递
   -> 记录结果
   -> 失败则重试 / 死信
```

这么做的好处：

- 不阻塞主业务响应
- 可重试
- 可观测
- 可手工补偿

## 七、Node 26 / ESM 投递示例

```js
import { signWebhook } from './sign-webhook.js';

export async function deliverWebhook({ targetUrl, path, payload, secret }) {
  const bodyText = JSON.stringify(payload);
  const timestamp = String(Date.now());
  const signature = signWebhook({
    method: 'POST',
    path,
    timestamp,
    bodyText
  }, secret);

  const response = await fetch(targetUrl, {
    method: 'POST',
    headers: {
      'content-type': 'application/json',
      'x-webhook-id': payload.eventId,
      'x-webhook-timestamp': timestamp,
      'x-webhook-signature': signature,
      'x-webhook-signature-method': 'HMAC-SHA256'
    },
    body: bodyText
  });

  return {
    status: response.status,
    ok: response.ok,
    body: await response.text()
  };
}
```

## 八、调用方验签示例

```js
import { createHash, createHmac, timingSafeEqual } from 'node:crypto';

function sha256Hex(text) {
  return createHash('sha256').update(text).digest('hex');
}

export function verifyWebhook({ method, path, timestamp, bodyText, signature }, secret) {
  const canonical = [
    method.toUpperCase(),
    path,
    timestamp,
    sha256Hex(bodyText)
  ].join('\n');

  const expected = createHmac('sha256', secret).update(canonical).digest('hex');
  const left = Buffer.from(expected, 'utf8');
  const right = Buffer.from(signature, 'utf8');

  if (left.length !== right.length) {
    return false;
  }

  return timingSafeEqual(left, right);
}
```

## 九、重试策略

推荐按结果分类：

- 2xx：成功
- 408 / 429 / 5xx：重试
- 大部分 4xx：不重试，转人工排查

重试时间示例：

```text
1m -> 5m -> 15m -> 1h -> 6h
```

此外要限制最大重试次数，避免无限积压。

## 十、Webhook 存储模型建议

### `webhook_endpoints`

- endpointId
- tenantId
- url
- secretCiphertext
- status

### `webhook_events`

- eventId
- eventType
- tenantId
- payload
- status
- nextRetryAt
- retryCount

### `webhook_deliveries`

- deliveryId
- eventId
- requestHeaders
- responseStatus
- responseBody
- durationMs

## 十一、平台治理建议

管理后台最好支持：

- 查看回调历史
- 查看失败原因
- 手工重发单个事件
- 旋转 webhook secret
- 测试回调地址

这些能力会显著降低排障和支持成本。

## 十二、本篇结论

Webhook 本质上是“平台向外推送事件”的能力，不是附带的小功能。要把它设计成：

- 事件化
- 可验签
- 可重试
- 可审计
- 可补偿

这样它才能长期稳定服务企业接入场景。
