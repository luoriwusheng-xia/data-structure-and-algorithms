# 05 - 日志、审计与可观测性

开放平台出问题时，第一现场通常不在代码，而在日志和监控里。

## 一、先区分三类日志

### 1. 访问日志

记录一次请求从进到出的基本过程：

- requestId
- method
- path
- statusCode
- duration
- tenantId
- appId

### 2. 业务日志

记录业务过程中的关键节点：

- 订单查询命中哪个数据源
- 支付发起调用了哪个下游
- Webhook 第几次重试

### 3. 审计日志

记录平台治理动作：

- 创建应用
- 旋转密钥
- 调整权限
- 修改白名单
- 手工重放回调

三类日志不能混为一类。

## 二、结构化日志是基本要求

不要依赖大量自由文本 `console.log('xxxx')`。

推荐输出 JSON 日志。

示例：

```json
{
  "level": "info",
  "time": "2026-07-01T08:00:00.000Z",
  "requestId": "req_001",
  "traceId": "trace_001",
  "tenantId": "tenant_001",
  "appId": "app_live_001",
  "method": "POST",
  "path": "/partner/v1/orders/query",
  "statusCode": 200,
  "durationMs": 23,
  "message": "request completed"
}
```

## 三、日志字段建议

开放平台建议统一保留这些字段：

| 字段 | 说明 |
|------|------|
| `time` | 日志时间 |
| `level` | 日志级别 |
| `requestId` | 单请求唯一标识 |
| `traceId` | 调用链标识 |
| `tenantId` | 租户标识 |
| `appId` | 应用标识 |
| `event` | 事件名 |
| `message` | 简短说明 |
| `durationMs` | 耗时 |
| `errorCode` | 平台错误码 |

## 四、最小日志封装示例

```js
import { getRequestContext } from '../core/request-context.js';

function write(level, message, extra = {}) {
  const context = getRequestContext();

  const log = {
    level,
    time: new Date().toISOString(),
    requestId: context.requestId ?? null,
    traceId: context.traceId ?? null,
    tenantId: context.tenantId ?? null,
    appId: context.appId ?? null,
    message,
    ...extra
  };

  process.stdout.write(`${JSON.stringify(log)}\n`);
}

export const logger = {
  info(message, extra) {
    write('info', message, extra);
  },
  warn(message, extra) {
    write('warn', message, extra);
  },
  error(message, extra) {
    write('error', message, extra);
  }
};
```

这不是完整日志系统，但已经比散乱输出强很多。

## 五、请求日志中间件

```js
import { logger } from '../infra/logger.js';

export function requestLogMiddleware(req, res, next) {
  const startedAt = Date.now();

  res.on('finish', () => {
    logger.info('request completed', {
      event: 'request.completed',
      method: req.method,
      path: req.originalUrl,
      statusCode: res.statusCode,
      durationMs: Date.now() - startedAt
    });
  });

  next();
}
```

## 六、鉴权日志和业务日志要分层

例如验签失败：

```js
logger.warn('signature verification failed', {
  event: 'auth.signature.failed',
  appId,
  path: req.path,
  reason: 'signature mismatch'
});
```

例如业务成功：

```js
logger.info('order query success', {
  event: 'order.query.success',
  orderNo,
  source: 'primary-db'
});
```

这样检索时更清晰。

## 七、审计日志设计

审计日志通常建议单独存储，不要只进普通应用日志。

典型字段：

- actorType：用户 / 系统 / 定时任务
- actorId
- targetType：app / credential / whitelist / webhook
- targetId
- action
- before
- after
- ip
- userAgent
- occurredAt

例如：

```json
{
  "actorType": "admin_user",
  "actorId": "u_001",
  "targetType": "credential",
  "targetId": "cred_009",
  "action": "credential.rotate",
  "occurredAt": "2026-07-01T08:00:00.000Z"
}
```

## 八、监控指标建议

日志负责还原现场，指标负责快速感知异常。

至少监控：

- QPS
- 成功率
- 4xx 比例
- 5xx 比例
- 平均耗时
- P95 / P99 耗时
- 限流命中次数
- 验签失败次数
- Webhook 重试次数

## 九、告警建议

开放平台通常至少要有下面这些告警：

- 5xx 突增
- 某租户请求异常暴涨
- 验签失败率异常
- Webhook 积压
- 回调失败率过高
- 下游依赖超时

## 十、敏感信息处理

日志里最容易出事故的不是“没有日志”，而是“日志过度”。

绝对不要直接打印：

- access token
- app secret
- 请求完整身份证号
- 银行卡全号
- 原始签名密钥

要么脱敏，要么摘要化。

## 十一、问题排查路径

一个成熟平台，排障时通常按这个顺序看：

1. 先看请求是否进入平台
2. 再看鉴权和签名是否通过
3. 再看幂等 / 限流是否命中
4. 再看业务服务执行过程
5. 再看下游依赖
6. 最后看异步回调和补偿

如果日志字段设计统一，这个过程会非常快。

## 十二、本篇结论

对外 API 平台的日志体系，目标不是“记录得多”，而是“能还原、能检索、能审计、能告警”。

如果你的平台未来要给企业长期接入，日志和可观测性必须在第一版就进入架构，而不是上线后再补。
