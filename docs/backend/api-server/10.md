# 10 - OpenTelemetry 接入与链路追踪

前面我们把日志、审计、请求上下文都搭起来了，但如果平台开始依赖多个服务、数据库、缓存、消息队列，仅靠日志已经不够。你还需要分布式链路追踪。

这一篇讨论如何把 OpenTelemetry 接到 Node.js API 平台里。

> 这里关于 NodeSDK、自动注入、OTLP exporter 的做法，参考了 OpenTelemetry JavaScript 官方文档；文中的 ESM 写法是在官方 CommonJS 示例基础上，按 Node 26 / ESM 风格转换后的等价写法。

## 一、为什么平台要接 OpenTelemetry

典型问题：

- 某个接口偶发超时，到底慢在平台本身还是下游服务？
- 某个租户投诉接口错误率升高，是限流命中、数据库慢查询，还是外部回调失败？
- 一次请求穿过 gateway、platform、db、redis、mq 之后，耗时到底分布在哪里？

这类问题用 trace 更适合回答。

## 二、平台里最值得追踪的链路

建议优先追踪：

- 入站 HTTP 请求
- 出站 HTTP 调用
- SQL 查询
- Redis 调用
- MQ 投递和消费
- Webhook 投递

不要一上来追所有东西，先覆盖最关键链路。

## 三、推荐接入方式

对于 Node.js 平台，建议优先使用：

- `NodeSDK`
- 自动注入 `getNodeAutoInstrumentations()`
- OTLP exporter
- `service.name` 资源属性

官方文档的核心要求之一是：SDK 要在业务代码加载前初始化。

## 四、基础依赖

```bash
pnpm add @opentelemetry/sdk-node
pnpm add @opentelemetry/auto-instrumentations-node
pnpm add @opentelemetry/exporter-trace-otlp-http
pnpm add @opentelemetry/resources
pnpm add @opentelemetry/semantic-conventions
```

## 五、最小 ESM 初始化示例

### `src/telemetry.js`

```js
import process from 'node:process';
import { NodeSDK } from '@opentelemetry/sdk-node';
import { OTLPTraceExporter } from '@opentelemetry/exporter-trace-otlp-http';
import { getNodeAutoInstrumentations } from '@opentelemetry/auto-instrumentations-node';
import { resourceFromAttributes } from '@opentelemetry/resources';
import { ATTR_SERVICE_NAME } from '@opentelemetry/semantic-conventions';

const traceExporter = new OTLPTraceExporter({
  url: process.env.OTEL_EXPORTER_OTLP_TRACES_ENDPOINT
    ?? 'http://localhost:4318/v1/traces'
});

const sdk = new NodeSDK({
  resource: resourceFromAttributes({
    [ATTR_SERVICE_NAME]: 'open-platform-api'
  }),
  traceExporter,
  instrumentations: [getNodeAutoInstrumentations()]
});

sdk.start();

process.on('SIGTERM', () => {
  sdk.shutdown()
    .finally(() => process.exit(0));
});
```

### `src/bootstrap.js`

```js
import './telemetry.js';
import './server.js';
```

然后用：

```bash
node src/bootstrap.js
```

重点不是文件名，而是 `telemetry.js` 必须先于应用代码加载。

## 六、自动注入能解决什么

自动注入通常可以覆盖：

- HTTP server
- HTTP client
- `fetch`
- 常见数据库驱动
- 常见框架和基础库

这意味着你不必为每个请求手写 span，就能先拿到一版链路图。

## 七、平台里仍然需要手工埋点的地方

自动注入不能替代业务语义。

建议手工补这些 span：

- 签名校验
- 幂等命中
- 权限判定
- Webhook 投递
- 配额扣减
- 调用模型或第三方支付

例如某条链路里，“为什么请求慢” 可能不是 HTTP 慢，而是“签名校验查凭证 + 权限判断 + Redis 幂等检查”叠加导致的。

## 八、手工创建 span 示例

```js
import { trace, SpanStatusCode } from '@opentelemetry/api';

const tracer = trace.getTracer('open-platform-auth');

export async function verifyRequestSignature(input) {
  return tracer.startActiveSpan('signature.verify', async (span) => {
    try {
      span.setAttribute('platform.app_id', input.appId);
      span.setAttribute('platform.signature_method', input.signatureMethod);

      const result = input.signature === 'expected-demo-signature';
      span.setAttribute('platform.signature_valid', result);

      if (!result) {
        span.setStatus({
          code: SpanStatusCode.ERROR,
          message: 'signature invalid'
        });
      }

      return result;
    } catch (error) {
      span.recordException(error);
      span.setStatus({
        code: SpanStatusCode.ERROR,
        message: error.message
      });
      throw error;
    } finally {
      span.end();
    }
  });
}
```

## 九、trace 和日志如何关联

你已经在前面的文章里做了 `requestId` / `traceId`。现在建议让日志也带上当前 trace 信息。

思路是：

- 在请求入口生成或继承 trace context
- 日志输出时附带 traceId / spanId
- 在日志平台中按 traceId 检索

这样可以实现：

- 先从告警进入某条 trace
- 再通过 traceId 检索同一请求的全部日志

## 十、OpenTelemetry Collector 的位置

推荐部署链路：

```text
Node.js API Platform
   -> OTLP HTTP
   -> OpenTelemetry Collector
   -> Jaeger / Tempo / Datadog / New Relic / Elastic
```

不要让业务应用直接对接一堆后端系统。Collector 可以统一做：

- 转发
- 采样
- 标签处理
- 后端切换

## 十一、采样建议

平台早期可以先全量采集低流量环境；生产环境通常要做采样。

建议：

- sandbox：高采样率
- production：按流量和成本调整
- 错误请求：尽量提高保留率

高价值接口，例如支付、计费、模型调用，可以单独提高采样策略。

## 十二、平台落地建议

第一阶段：

- 先接 `NodeSDK`
- 先开自动注入
- 先打通 OTLP 到 Collector

第二阶段：

- 补关键业务 span
- 让日志关联 traceId
- 补 dashboard 和告警

第三阶段：

- 按租户、接口、下游服务做分组观测
- 建立高价值链路的 SLO

## 十三、本篇结论

OpenTelemetry 的价值不在“多一个监控组件”，而在于它让你能精确回答：

- 一次请求经过了哪些环节
- 每个环节耗时多少
- 错误发生在哪一层

对于要长期服务企业接入的 API 平台，这种能力是必要的，不是锦上添花。
