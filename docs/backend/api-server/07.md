# 07 - API Key 发放与接入流程

前面已经讨论了鉴权模型，这一篇把“平台如何把调用能力交付给第三方”讲清楚。很多接口平台不是死在签名算法上，而是死在接入流程混乱、凭证发放失控。

## 一、API Key 不是一串字符串，而是一套接入资产

一个成熟的平台在给客户开通接入能力时，通常要一起发放或确认这些东西：

- tenantId
- appId
- apiKey
- keyId
- secret / public key 交换方式
- sandbox 域名
- production 域名
- 签名算法
- 白名单 IP 规则
- 回调地址规则

因此，平台侧不要把“创建 API Key”理解成单表插入一行数据，而要把它当成一次正式的接入开通动作。

## 二、推荐的接入对象模型

推荐这样组织：

```text
Tenant
  └── App
        ├── Environment (sandbox / production)
        ├── Credential
        ├── Permission Set
        └── Quota Policy
```

这样做的原因：

- 一个租户可能有多个应用
- 同一个应用会区分测试和生产
- 不同环境必须使用不同凭证
- 权限和配额不应直接写死在凭证上

## 三、API Key 发放流程

推荐流程如下：

1. 平台创建租户
2. 为租户创建应用
3. 选择环境：sandbox 或 production
4. 生成 `appId`、`apiKey`、`keyId`
5. 生成或登记 secret / public key
6. 绑定权限集和限流策略
7. 生成接入指引与示例代码
8. 记录审计日志

如果你的平台是企业客户接入，不建议允许用户只拿到一组永不过期的 secret 然后长期使用。

## 四、推荐字段设计

### 1. 应用表 `apps`

| 字段 | 说明 |
|------|------|
| `app_id` | 应用唯一标识 |
| `tenant_id` | 所属租户 |
| `app_name` | 应用名称 |
| `environment` | `sandbox` / `production` |
| `status` | 启用、禁用、冻结 |
| `created_at` | 创建时间 |

### 2. 凭证表 `credentials`

| 字段 | 说明 |
|------|------|
| `credential_id` | 凭证主键 |
| `app_id` | 所属应用 |
| `key_id` | 密钥标识 |
| `api_key` | 对外公开标识 |
| `secret_ciphertext` | 加密后的 secret |
| `algorithm` | `HMAC-SHA256` / `ED25519` |
| `status` | 激活、过期、吊销 |
| `expires_at` | 过期时间 |

`api_key` 负责身份识别，`secret` 负责签名，不要混用一个字段。

## 五、生成凭证的基本原则

### 1. `appId` 可读，secret 不可读

例如：

```text
app_live_29fce2e0
key_live_01J123XYZ
sk_live_v2x8...
```

### 2. secret 只展示一次

平台创建 secret 后，通常只在创建成功的那一刻展示一次。之后只保留：

- 加密后的密文
- 指纹
- 最后 4 位

### 3. 生产和测试必须分离

这不是体验问题，而是风险控制问题。

## 六、接入文档里必须明确的内容

任何一组新凭证发出去时，文档里至少要说明：

- 调用域名
- 必需请求头
- 签名算法
- canonical string 规则
- 时间戳允许误差
- nonce 是否要求唯一
- 错误码说明
- 限流策略

平台侧的“接入成功率”，很大程度取决于这部分是否足够具体。

## 七、一个最小发放示例

下面给一个 Node 26 / ESM 风格的示例，演示平台如何生成一组 API 凭证。

```js
import { randomBytes, randomUUID } from 'node:crypto';

function toBase62(buffer) {
  return buffer.toString('base64url');
}

export function issueCredential({ tenantId, environment }) {
  const appId = `app_${environment}_${randomUUID().slice(0, 8)}`;
  const keyId = `key_${environment}_${Date.now()}`;
  const apiKey = `ak_${environment}_${toBase62(randomBytes(18))}`;
  const secret = `sk_${environment}_${toBase62(randomBytes(32))}`;

  return {
    tenantId,
    appId,
    keyId,
    apiKey,
    secret,
    algorithm: 'HMAC-SHA256'
  };
}
```

在真实系统中，这里生成出来的 `secret` 应当立即加密后入库。

## 八、接入引导建议

平台控制台可以把接入流程拆成 4 步：

1. 创建应用
2. 获取 API 凭证
3. 配置 IP 白名单 / 回调地址
4. 调试 sandbox 示例请求

如果你的平台有管理后台，这一步体验做好，能显著减少接入支持成本。

## 九、常见错误

### 1. 一个租户共用一套永久凭证

问题：

- 无法精细控制应用边界
- 无法隔离环境
- 无法平滑轮换

### 2. secret 可以在后台反复明文查看

风险很高。正确做法是只显示一次，后台只保留加密密文与指纹。

### 3. 没有接入审计

谁创建了哪个应用、什么时候发放了哪套凭证、何时启用生产环境，这些都必须有审计记录。

## 十、本篇结论

API Key 发放本质上是“开放平台接入开通流程”的一部分。设计重点不在随机串生成，而在：

- 接入对象边界清晰
- 环境隔离明确
- secret 生命周期可治理
- 文档与引导可执行
