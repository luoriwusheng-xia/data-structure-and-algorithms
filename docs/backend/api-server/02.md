# 02 - Node 26 + ESM 工程骨架

这一篇开始落地。目标是搭一个适合开放平台场景的 Node.js 26 工程骨架，而不是只写一个 `app.js`。

## 一、为什么统一使用 ESM

本专题统一使用 ESM，原因很直接：

- 语义更现代
- 更贴近浏览器与标准 JavaScript
- 与 Node 26 的现代能力配合更自然
- `import` / `export`、top-level `await`、`node:` 前缀都更清晰

示例：

```js
import http from 'node:http';
import { randomUUID } from 'node:crypto';
import process from 'node:process';
```

## 二、推荐目录结构

```text
src/
  app.js
  server.js
  config/
    index.js
  core/
    errors.js
    response.js
    request-context.js
  middleware/
    auth.js
    signature.js
    rate-limit.js
    idempotency.js
    request-log.js
  modules/
    open-platform/
      open-platform.controller.js
      open-platform.service.js
      open-platform.repository.js
  infra/
    db.js
    redis.js
    logger.js
  routes/
    public.routes.js
    partner.routes.js
```

这个结构的重点不是“分层漂亮”，而是职责清楚：

- `core`：平台通用能力
- `middleware`：接入治理
- `modules`：领域业务
- `infra`：数据库、缓存、日志等外部依赖

## 三、启动入口拆分

建议把 `app` 和 `server` 分开。

### `src/app.js`

```js
import express from 'express';
import { requestLogMiddleware } from './middleware/request-log.js';
import { authMiddleware } from './middleware/auth.js';
import { signatureMiddleware } from './middleware/signature.js';
import { publicRoutes } from './routes/public.routes.js';
import { partnerRoutes } from './routes/partner.routes.js';

export function createApp() {
  const app = express();

  app.use(express.json({ limit: '1mb' }));
  app.use(requestLogMiddleware);

  app.use('/public', publicRoutes);
  app.use('/partner', authMiddleware, signatureMiddleware, partnerRoutes);

  return app;
}
```

### `src/server.js`

```js
import { createServer } from 'node:http';
import process from 'node:process';
import { createApp } from './app.js';

const app = createApp();
const server = createServer(app);
const port = Number(process.env.PORT ?? 3000);

server.listen(port, () => {
  console.log(`API platform listening on http://localhost:${port}`);
});
```

这么做的目的：

- `app.js` 负责组装应用
- `server.js` 负责监听端口
- 测试时更容易单独加载 app

## 四、配置管理

不要把环境变量散落到各个文件里。

### `src/config/index.js`

```js
import process from 'node:process';

function readRequired(name) {
  const value = process.env[name];
  if (!value) {
    throw new Error(`Missing required env: ${name}`);
  }
  return value;
}

export const config = {
  app: {
    env: process.env.NODE_ENV ?? 'development',
    port: Number(process.env.PORT ?? 3000)
  },
  auth: {
    issuer: process.env.API_ISSUER ?? 'open-platform',
    accessTokenSecret: readRequired('ACCESS_TOKEN_SECRET')
  },
  signature: {
    skewMs: Number(process.env.SIGNATURE_SKEW_MS ?? 5 * 60 * 1000)
  },
  log: {
    level: process.env.LOG_LEVEL ?? 'info'
  }
};
```

原则：

- 必填项在启动阶段就失败
- 所有配置统一导出
- 业务代码不要直接读取 `process.env`

## 五、统一错误与响应结构

开放平台不应该让调用方猜测错误格式。

### `src/core/errors.js`

```js
export class AppError extends Error {
  constructor(message, options = {}) {
    super(message);
    this.name = 'AppError';
    this.code = options.code ?? 'INTERNAL_ERROR';
    this.status = options.status ?? 500;
    this.details = options.details ?? null;
  }
}
```

### `src/core/response.js`

```js
export function ok(data, meta = {}) {
  return {
    success: true,
    data,
    meta
  };
}

export function fail(error) {
  return {
    success: false,
    error: {
      code: error.code,
      message: error.message,
      details: error.details ?? null
    }
  };
}
```

建议平台统一响应体：

```json
{
  "success": false,
  "error": {
    "code": "SIGNATURE_INVALID",
    "message": "signature is invalid",
    "details": null
  }
}
```

## 六、请求上下文

企业平台必须把 `requestId`、`tenantId`、`appId`、`traceId` 绑定到当前请求上下文。

Node 可以用 `AsyncLocalStorage` 做这件事。

### `src/core/request-context.js`

```js
import { AsyncLocalStorage } from 'node:async_hooks';
import { randomUUID } from 'node:crypto';

const storage = new AsyncLocalStorage();

export function requestContextMiddleware(req, res, next) {
  const requestId = req.headers['x-request-id'] ?? randomUUID();
  const traceId = req.headers['x-trace-id'] ?? requestId;

  storage.run(
    {
      requestId,
      traceId,
      tenantId: null,
      appId: null
    },
    () => next()
  );
}

export function getRequestContext() {
  return storage.getStore() ?? {};
}
```

这套上下文后面会被日志、鉴权、审计统一使用。

## 七、Node 26 常用现代能力

### 1. `fetch`

服务端调用下游时可直接用标准 `fetch`：

```js
const response = await fetch('https://partner.example.com/callback', {
  method: 'POST',
  headers: {
    'content-type': 'application/json'
  },
  body: JSON.stringify({ status: 'ok' })
});
```

### 2. `AbortController`

必须给外部调用设置超时：

```js
const controller = new AbortController();
const timeout = setTimeout(() => controller.abort(), 3000);

try {
  const response = await fetch('https://example.com/api', {
    signal: controller.signal
  });
  return await response.json();
} finally {
  clearTimeout(timeout);
}
```

### 3. `node:crypto`

用于：

- 生成 requestId
- HMAC 签名
- 公私钥签名校验
- token secret 相关操作

## 八、中间件顺序建议

顺序非常重要。

```text
request-context
  -> request-log
  -> body parse
  -> auth
  -> signature verify
  -> idempotency
  -> rate limit
  -> controller
  -> error handler
```

理由：

- 先建立上下文，后面日志才能统一拿到 requestId
- 验签应发生在真正进入业务之前
- 幂等与限流属于平台治理层，而不是业务层

## 九、Controller / Service / Repository 边界

推荐严格约束：

### Controller

只做：

- 收参数
- 调 service
- 返回响应

### Service

只做：

- 业务规则
- 多仓储编排
- 下游调用

### Repository

只做：

- 数据存取

不要把签名校验、日志拼接、SQL、业务判断全塞到一个文件里。

## 十、一个最小控制器示例

```js
import { Router } from 'express';
import { ok } from '../core/response.js';

export const partnerRoutes = Router();

partnerRoutes.post('/v1/orders/query', async (req, res) => {
  const result = {
    orderNo: req.body.orderNo,
    status: 'SUCCESS'
  };

  res.json(ok(result));
});
```

## 十一、工程落地建议

第一版平台，不建议一上来引入过多框架魔法。你需要先把下面这些做扎实：

- 清晰目录结构
- 统一响应格式
- 请求上下文
- 鉴权与验签中间件
- 结构化日志
- 错误处理

这些是开放平台的骨架。
