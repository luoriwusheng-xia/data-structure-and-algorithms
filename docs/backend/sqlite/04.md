## 1. 基础概念篇

### Q1：SQLite 是什么？它与 MySQL 最大的区别是什么？

**答案**：

SQLite 是一个嵌入式关系型数据库引擎，没有独立的服务器进程，数据库存储在单个文件中。

核心区别：

| 维度 | SQLite | MySQL |
|------|--------|-------|
| 架构 | 嵌入式，进程内 | C/S 架构 |
| 部署 | 零配置，一个文件 | 需安装、配置、启动服务 |
| 并发 | 写操作串行 | 支持高并发读写 |
| 网络 | 不支持远程访问 | 支持网络连接 |
| 适用 | 单机应用、移动端 | 大型 Web 应用 |

### Q2：SQLite 的数据类型系统是什么？有哪些存储类型？

**答案**：

SQLite 采用**动态类型系统（Type Affinity）**，定义类型只是建议，实际存储取决于值的类型。

5 种存储类型：

```sql
NULL      -- 空值
INTEGER   -- 整型（1-8 字节）
REAL      -- 浮点数（IEEE 754）
TEXT      -- 文本字符串
BLOB      -- 二进制数据
```

**陷阱**：

```sql
CREATE TABLE t (a INTEGER);
INSERT INTO t VALUES ('hello');  -- 能插入！存储为 TEXT
INSERT INTO t VALUES (3.14);     -- 能插入！存储为 REAL
```

### Q3：SQLite 如何存储 Boolean 和日期时间？

**答案**：

**Boolean**：没有原生类型，用 INTEGER，`0` = false，`1` = true。

```sql
CREATE TABLE users (
    is_active INTEGER DEFAULT 0 CHECK (is_active IN (0, 1))
);
```

**日期时间**：没有 DATE/TIME 类型，三种存储方式：

```sql
-- 方式 1：TEXT，ISO8601 格式（推荐）
'2024-01-15 09:30:00'

-- 方式 2：INTEGER，Unix 时间戳
1705312200

-- 方式 3：REAL，Julian Day
2460322.5
```

内置日期函数：`date()`, `time()`, `datetime()`, `julianday()`, `strftime()`

### Q4：SQLite 的 `AUTOINCREMENT` 和 `INTEGER PRIMARY KEY` 有什么区别？

**答案**：

```sql
-- 方式 1：INTEGER PRIMARY KEY（推荐，ROWID 别名）
CREATE TABLE t1 (id INTEGER PRIMARY KEY, name TEXT);
-- 自动使用 ROWID，新值 = max(ROWID) + 1
-- 效率高，删除后不重新利用 ID

-- 方式 2：INTEGER PRIMARY KEY AUTOINCREMENT
CREATE TABLE t2 (id INTEGER PRIMARY KEY AUTOINCREMENT, name TEXT);
-- 使用独立的序列计数器
-- 保证 ID 单调递增，即使删除后也不会复用
-- 性能略低，因为需要维护 sqlite_sequence 表
-- 适用于需要保证 ID 不重复的场景（如分布式 ID 生成）
```

### Q5：SQLite 支持哪些 JOIN？哪些不支持？

**答案**：

**支持**：INNER JOIN、LEFT JOIN、CROSS JOIN

**不支持**：RIGHT JOIN、FULL OUTER JOIN

**变通方案**：

```sql
-- RIGHT JOIN → 交换表位置用 LEFT JOIN
-- A RIGHT JOIN B = B LEFT JOIN A

-- FULL JOIN → UNION 两个 LEFT JOIN
SELECT * FROM a LEFT JOIN b ON a.id = b.id
UNION
SELECT * FROM b LEFT JOIN a ON a.id = b.id;
```

## 2. SQL 语法篇

### Q6：写出 SQLite 中实现 Upsert（插入或更新）的 SQL。

**答案**：

```sql
-- SQLite 3.24.0+ 支持 ON CONFLICT
INSERT INTO users (id, name, email, updated_at)
VALUES (1, '张三', 'new@example.com', datetime('now'))
ON CONFLICT(id) DO UPDATE SET
    name = excluded.name,
    email = excluded.email,
    updated_at = excluded.updated_at;

-- DO NOTHING：冲突时不操作
INSERT INTO users (id, name) VALUES (1, '张三')
ON CONFLICT(id) DO NOTHING;

-- 指定约束名
INSERT INTO users (email, name) VALUES ('a@b.com', '张三')
ON CONFLICT(email) DO UPDATE SET name = excluded.name;
```

### Q7：SQLite 中如何分页查询？与 MySQL 有何不同？

**答案**：

```sql
-- SQLite（两种方式）
SELECT * FROM users LIMIT 10 OFFSET 20;   -- 标准方式
SELECT * FROM users LIMIT 20, 10;          -- SQLite 特有简写（OFFSET, LIMIT）

-- MySQL
SELECT * FROM users LIMIT 20, 10;          -- 与 SQLite 相同
SELECT * FROM users LIMIT 10 OFFSET 20;    -- 也支持

-- 分页优化（避免深度分页性能问题）
-- 方式 1：使用覆盖索引
SELECT * FROM users WHERE id > ? ORDER BY id LIMIT 10;

-- 方式 2：先查 ID 再 JOIN
SELECT u.* FROM users u
JOIN (SELECT id FROM users ORDER BY id LIMIT 10000, 10) t ON u.id = t.id;
```

### Q8：递归 CTE 有什么用？给出一个实际例子。

**答案**：

递归 CTE 用于查询树形/层级数据，如组织架构、文件目录、分类树。

```sql
-- 员工表（id, name, manager_id）
CREATE TABLE employees (
    id INTEGER PRIMARY KEY,
    name TEXT,
    manager_id INTEGER REFERENCES employees(id)
);

-- 查询某经理的所有下属（递归）
WITH RECURSIVE subordinates AS (
    -- 锚定成员：起始经理
    SELECT id, name, manager_id, 0 AS level
    FROM employees WHERE id = 1

    UNION ALL

    -- 递归成员：找下属
    SELECT e.id, e.name, e.manager_id, s.level + 1
    FROM employees e
    INNER JOIN subordinates s ON e.manager_id = s.id
)
SELECT * FROM subordinates ORDER BY level;

-- 查询某员工的所有上级（反向递归）
WITH RECURSIVE managers AS (
    SELECT id, name, manager_id, 0 AS level
    FROM employees WHERE id = 5

    UNION ALL

    SELECT e.id, e.name, e.manager_id, m.level + 1
    FROM employees e
    INNER JOIN managers m ON e.id = m.manager_id
)
SELECT * FROM managers ORDER BY level;
```

### Q9：SQLite 中 `EXISTS` 和 `IN` 哪个更好？为什么？

**答案**：

```sql
-- EXISTS 通常优于 IN（尤其是子查询结果大时）

-- IN：先执行子查询，构造结果集，再比较
SELECT * FROM users WHERE id IN (SELECT user_id FROM orders WHERE amount > 100);

-- EXISTS：找到匹配即停止，短路求值
SELECT * FROM users u WHERE EXISTS (
    SELECT 1 FROM orders o WHERE o.user_id = u.id AND o.amount > 100
);

-- EXISTS 优势：
-- 1. 不需要构建临时结果集
-- 2. 遇到匹配即返回，性能更好
-- 3. 处理 NULL 更安全

-- NOT EXISTS vs NOT IN 的区别
SELECT * FROM users WHERE id NOT IN (SELECT user_id FROM orders);
-- 危险：如果子查询中有 NULL，结果全为 FALSE！

SELECT * FROM users u WHERE NOT EXISTS (
    SELECT 1 FROM orders o WHERE o.user_id = u.id
);
-- 安全：NOT EXISTS 不受 NULL 影响
```

### Q10：SQLite 如何处理 JSON 数据？

**答案**：

```sql
-- 3.38.0+ 内置 JSON 支持
CREATE TABLE events (id INTEGER PRIMARY KEY, data TEXT);

-- 存储 JSON
INSERT INTO events (data) VALUES ('{"user": "张三", "action": "login", "ip": "192.168.1.1"}');

-- 提取字段
SELECT json_extract(data, '$.user') FROM events;

-- 过滤
SELECT * FROM events WHERE json_extract(data, '$.action') = 'login';

-- 修改
UPDATE events SET data = json_set(data, '$.count', 10);

-- 数组操作
UPDATE events SET data = json_array_insert(data, '$.tags', 'new');

-- 展开数组（每行一个元素）
SELECT e.id, json_each.value AS tag
FROM events e, json_each(e.data, '$.tags');
```

## 3. 进阶特性篇

### Q11：什么是 WAL 模式？为什么要用它？

**答案**：

WAL（Write-Ahead Logging）是 SQLite 的一种日志模式。

**传统模式（DELETE）**：修改数据前先拷贝原页到 journal 文件，修改主数据库，提交后删除 journal。读和写互相阻塞。

**WAL 模式**：修改追加到 `-wal` 文件，读取直接从主数据库文件（不受写入影响），定期 checkpoint 合并回主库。

```sql
PRAGMA journal_mode = WAL;
```

**WAL 优势**：
- 读不阻塞写，写不阻塞读
- 并发性能大幅提升
- 崩溃恢复更快

**WAL 劣势**：
- 需要定期 checkpoint
- WAL 文件可能变大
- 不支持内存映射（mmap）

### Q12：SQLite 的事务隔离级别是什么？

**答案**：

SQLite 默认隔离级别是 **SERIALIZABLE**（可串行化）。

```sql
-- SQLite 支持三种 BEGIN 模式
BEGIN DEFERRED;    -- 默认，第一次访问时获取锁
BEGIN IMMEDIATE;   -- 立即获取 RESERVED 锁，适合写操作
BEGIN EXCLUSIVE;   -- 立即获取 EXCLUSIVE 锁，完全独占
```

**锁的五种状态**：

```
UNLOCKED → SHARED → RESERVED → PENDING → EXCLUSIVE
```

### Q13：什么是 FTS5？它如何实现全文搜索？

**答案**：

FTS5（Full-Text Search 5）是 SQLite 的全文搜索扩展，基于**倒排索引**实现。

```sql
-- 创建 FTS5 虚拟表
CREATE VIRTUAL TABLE docs USING fts5(title, content);

-- 插入数据
INSERT INTO docs (title, content) VALUES ('SQLite 教程', 'SQLite 是一个嵌入式数据库');

-- 搜索
SELECT * FROM docs WHERE docs MATCH 'SQLite';
SELECT * FROM docs WHERE title MATCH '教程';

-- 排除
SELECT * FROM docs WHERE docs MATCH '数据库 -MySQL';

-- 短语
SELECT * FROM docs WHERE docs MATCH '"嵌入式数据库"';

-- 前缀
SELECT * FROM docs WHERE docs MATCH 'Sql*';
```

**FTS5 内部结构**：
- 倒排索引：词 → 文档 ID 列表
- 支持相关性排序（`rank`）
- 支持高亮显示（`highlight()`, `snippet()`）

### Q14：SQLite 的索引是什么结构？

**答案**：

SQLite 使用 **B-tree**（B+ 树变体）存储索引。

**特点**：
- 所有索引都是 B-tree
- 聚集索引：`INTEGER PRIMARY KEY` 就是 ROWID，数据按 ROWID 顺序存储
- 非聚集索引：叶子节点存储 ROWID，需要通过 ROWID 回表查数据

**最左前缀原则**：

```sql
CREATE INDEX idx_age_name ON users(age, name);
-- 有效：WHERE age = 20
-- 有效：WHERE age = 20 AND name = '张三'
-- 无效：WHERE name = '张三'（跳过了 age）
```

---

### Q15：什么是覆盖索引（Covering Index）？

**答案**：

覆盖索引是指查询所需的所有列都在索引中，不需要回表查数据。

```sql
CREATE INDEX idx_cover ON orders(user_id, status, amount);

-- 覆盖索引查询（只需读索引，不回表）
SELECT user_id, status, amount FROM orders WHERE user_id = 1;
-- EXPLAIN QUERY PLAN 显示：USING COVERING INDEX idx_cover

-- 非覆盖索引查询（需要回表）
SELECT user_id, status, amount, created_at FROM orders WHERE user_id = 1;
-- created_at 不在索引中，需要回表
```

---

### Q16：部分索引（Partial Index）是什么？有什么用？

**答案**：

部分索引只给满足条件的行建索引，节省空间。

```sql
-- 只给活跃用户建索引
CREATE INDEX idx_active ON users(last_login) WHERE is_active = 1;

-- 只给未完成的订单建索引
CREATE INDEX idx_pending ON orders(created_at) WHERE status = 'pending';

-- 只给金额大的订单建索引
CREATE INDEX idx_large ON orders(amount) WHERE amount > 10000;
```

**优势**：
- 索引更小，维护更快
- 查询优化器可以自动选择

### Q17：SQLite 的窗口函数有哪些？给一个实际例子。

**答案**：

SQLite 3.25+ 支持窗口函数。

```sql
-- 排名类：ROW_NUMBER(), RANK(), DENSE_RANK(), NTILE()
-- 聚合类：SUM(), AVG(), COUNT(), MAX(), MIN() OVER ()
-- 取值类：FIRST_VALUE(), LAST_VALUE(), LAG(), LEAD()

-- 实际例子：计算每门课程的学生排名
WITH ranked AS (
    SELECT
        student_name,
        course,
        score,
        RANK() OVER (PARTITION BY course ORDER BY score DESC) AS rank_in_course,
        AVG(score) OVER (PARTITION BY course) AS course_avg,
        score - AVG(score) OVER (PARTITION BY course) AS diff_from_avg
    FROM scores
)
SELECT * FROM ranked WHERE rank_in_course <= 3;  -- 每科前 3 名

-- 实际例子：同比环比计算
SELECT
    month,
    sales,
    LAG(sales, 1) OVER (ORDER BY month) AS last_month,
    LAG(sales, 12) OVER (ORDER BY month) AS last_year,
    ROUND((sales - LAG(sales, 1) OVER (ORDER BY month)) * 100.0 / LAG(sales, 1) OVER (ORDER BY month), 2) AS mom_growth
FROM monthly_sales;
```

## 4. 性能优化篇

### Q18：SQLite 插入大量数据如何优化？

**答案**：

```javascript
const db = require('better-sqlite3')('app.db');

// 1. 使用事务（最重要！提高 100 倍以上）
const insertMany = db.transaction((rows) => {
    const stmt = db.prepare('INSERT INTO users (name) VALUES (?)');
    for (const row of rows) stmt.run(row);
});

// 2. 关闭同步（导入时临时使用）
db.pragma('synchronous = OFF');
db.pragma('journal_mode = MEMORY');

// 3. 预编译语句
const stmt = db.prepare('INSERT INTO users (name) VALUES (?)');
for (let i = 0; i < 100000; i++) {
    stmt.run(`user_${i}`);
}

// 4. 批量插入（单条 SQL 多个 VALUES）
const stmt2 = db.prepare('INSERT INTO users (name) VALUES (?)');
// 或用 generate_series
INSERT INTO users (name)
SELECT 'user_' || value FROM generate_series(1, 100000);
```

### Q19：如何查看和优化查询执行计划？

**答案**：


### Q20：数据库文件越来越大怎么办？

**答案**：

```sql
-- 1. 执行 VACUUM（重建数据库，回收空间）
VACUUM;
VACUUM INTO 'new.db';  -- 导出到新文件

-- 2. 启用 auto_vacuum
PRAGMA auto_vacuum = INCREMENTAL;
PRAGMA incremental_vacuum(1000);

-- 3. 清理 WAL
PRAGMA wal_checkpoint(TRUNCATE);

-- 4. 定期删除旧数据
DELETE FROM logs WHERE created_at < date('now', '-90 days');
VACUUM;

-- 5. 检查是否有大字段（BLOB）
SELECT name, pageno FROM dbstat WHERE aggregate = TRUE ORDER BY pageno DESC;
```

### Q21：`PRAGMA` 有哪些常用的调优参数？

**答案**：

```sql
PRAGMA journal_mode = WAL;           -- WAL 模式
PRAGMA synchronous = NORMAL;         -- 同步模式（NORMAL/FULL/OFF）
PRAGMA cache_size = 10000;           -- 缓存页面数
PRAGMA mmap_size = 30000000000;      -- 内存映射大小
PRAGMA temp_store = MEMORY;          -- 临时表存储位置
PRAGMA foreign_keys = ON;            -- 启用外键（默认关闭！）
PRAGMA busy_timeout = 5000;          -- 锁等待超时（毫秒）
PRAGMA optimize;                     -- 运行优化器
PRAGMA analysis_limit = 1000;        -- ANALYZE 采样限制
```

## 5. 实战场景篇

### Q22：SQLite 适合什么场景？不适合什么场景？

**答案**：

**适合**：
- 桌面/移动应用本地存储

- 需要远程网络访问
- TB 级大数据量
- 多主/分布式部署
- 复杂用户权限管理

### Q25：SQLite 的备份方案有哪些？

**答案**：

```javascript
const db = require('better-sqlite3')('app.db');

// 方案 1：热备份（在线备份，不中断服务）
db.backup('backup.db');

// 方案 2：文件复制（WAL 模式下需要先 checkpoint）
db.pragma('wal_checkpoint(TRUNCATE)');
const fs = require('fs');
fs.copyFileSync('app.db', 'backup.db');

// 方案 3：导出 SQL
db.exec(".dump");  // 在命令行中

// 方案 4：使用 Litestream（云备份）
// Litestream 持续将 WAL 复制到 S3/云存储
// 支持时间点恢复

// 方案 5：定时脚本
function scheduledBackup() {
    const timestamp = new Date().toISOString().replace(/[:.]/g, '-');
    db.backup(`backups/backup_${timestamp}.db`);
}
setInterval(scheduledBackup, 24 * 60 * 60 * 1000);  // 每天备份
```

## 6. 架构设计篇

### Q26：SQLite 能否支持多进程访问？如何设计？

**答案**：

可以，但需要 WAL 模式 + 适当的锁策略。

```javascript
// 每个进程使用独立的连接
const Database = require('better-sqlite3');

// 进程 1（写进程）
const writer = new Database('app.db');
writer.pragma('journal_mode = WAL');
writer.pragma('busy_timeout = 10000');

// 进程 2（读进程）
const reader = new Database('app.db');
reader.pragma('journal_mode = WAL');
reader.pragma('busy_timeout = 10000');

// 设计建议：
// 1. WAL 模式是必须的
// 2. 设置 busy_timeout 处理锁等待
// 3. 读进程可以并发，写进程串行
// 4. 考虑使用队列序列化写操作
// 5. 高并发场景考虑用 Redis 做写入队列
```

### Q27：SQLite + Node.js 项目的数据库连接如何管理？

**答案**：

```javascript
// 单例模式管理连接
const Database = require('better-sqlite3');

let dbInstance = null;

function getDb() {
    if (!dbInstance) {
        dbInstance = new Database(process.env.DB_PATH || 'app.db');
        dbInstance.pragma('journal_mode = WAL');
        dbInstance.pragma('foreign_keys = ON');
        dbInstance.pragma('busy_timeout = 5000');
    }
    return dbInstance;
}

function closeDb() {
    if (dbInstance) {
        dbInstance.close();
        dbInstance = null;
    }
}

// 使用
const db = getDb();

// 注意：
// 1. SQLite 不需要连接池（单文件，一个连接即可）
```

### Q28：如何实现 SQLite 的数据库迁移？

**答案**：

```javascript
class MigrationManager {
    constructor(db) {
        this.db = db;
        this.db.exec(`
            CREATE TABLE IF NOT EXISTS __migrations (
                version INTEGER PRIMARY KEY,
                name TEXT,
                applied_at TEXT DEFAULT (datetime('now'))
            )
        `);
    }

    migrate(migrations) {
        const currentVersion = this.db.prepare(
            'SELECT MAX(version) AS v FROM __migrations'
        ).get().v || 0;

        for (const migration of migrations) {
            if (migration.version <= currentVersion) continue;

            this.db.transaction(() => {
                this.db.exec(migration.up);
                this.db.prepare(
                    'INSERT INTO __migrations (version, name) VALUES (?, ?)'
                ).run(migration.version, migration.name);
            })();
        }
    }
}

// 使用
const migrations = [
    {
        version: 1,
        name: 'initial_schema',
        up: `CREATE TABLE users (id INTEGER PRIMARY KEY, name TEXT)`,
    },
    {
        version: 2,
        name: 'add_email',
        up: `ALTER TABLE users ADD COLUMN email TEXT`,
    },
];

const manager = new MigrationManager(db);
manager.migrate(migrations);
```

## 7. 源码原理篇

### Q29：SQLite 的 B-tree 索引是如何工作的？

**答案**：

SQLite 使用 B+ 树变体存储所有数据：

**表 B-tree**（`SQLITE_BTREE_INTKEY`）：
- 内部节点：仅存储键值（ROWID）用于导航
- 叶子节点：存储完整的行数据
- `INTEGER PRIMARY KEY` 的表，数据按 ROWID 排序存储

**索引 B-tree**（`SQLITE_BTREE_ZERODATA`）：
- 内部节点：存储索引键 + ROWID
- 叶子节点：存储索引键 + ROWID（指向表数据）
- 非覆盖索引查询需要"回表"

**页结构**：
- 每个页（默认 4096 字节）
- 页头 + 单元指针数组 + 单元内容
- 空闲空间管理

### Q30：SQLite 的事务实现原理是什么？

**答案**：

SQLite 使用 **WAL（Write-Ahead Logging）** 或 **Rollback Journal** 实现 ACID：

**Rollback Journal 模式**：
1. 获取 SHARED 锁
2. 拷贝要修改的页到 journal 文件
3. 获取 RESERVED 锁
4. 修改数据库页
5. 获取 EXCLUSIVE 锁
6. 写入 journal 文件头（标记有效）
7. 释放 SHARED 锁
8. 清除 journal 文件

**WAL 模式**：
1. 修改追加到 WAL 文件末尾
2. WAL 文件头部记录"commit"标记
3. 读取时先检查 WAL，再检查主数据库
4. 定期 checkpoint：将 WAL 内容合并回主库

### Q31：SQLite 如何保证 ACID？

**答案**：

- **Atomicity（原子性）**：journal/WAL 机制，失败时回滚
- **Consistency（一致性）**：外键约束、CHECK 约束、触发器
- **Isolation（隔离性）**：锁机制 + WAL 的 snapshot 读
- **Durability（持久性）**：`fsync()` 刷盘（`PRAGMA synchronous` 控制）

```
同步模式：
FULL（默认）：每次 COMMIT 都 fsync，最安全
NORMAL：WAL 模式下每 COMMIT fsync，非 WAL 时不保证
OFF：不 fsync，最快但可能丢失最近事务
```

### Q34：手写一个递归 CTE，查询树形菜单。

```sql
CREATE TABLE menu (
    id INTEGER PRIMARY KEY,
    name TEXT,
    parent_id INTEGER,
    sort_order INTEGER
);

-- 查询某个菜单的所有子菜单（层级展开）
WITH RECURSIVE menu_tree AS (
    -- 锚定：从根菜单开始
    SELECT id, name, parent_id, sort_order, 0 AS level,
           CAST(name AS TEXT) AS path
    FROM menu
    WHERE parent_id IS NULL

    UNION ALL

    -- 递归：找子菜单
    SELECT m.id, m.name, m.parent_id, m.sort_order, mt.level + 1,
           mt.path || ' > ' || m.name
    FROM menu m
    INNER JOIN menu_tree mt ON m.parent_id = mt.id
)
SELECT
    REPEAT('  ', level) || name AS indented_name,
    path,
    level
FROM menu_tree
ORDER BY path;
```

### Q35：手写 Node.js 封装，实现一个带连接池的 SQLite 查询工具。

```javascript
// 注意：SQLite 实际上不需要连接池（单文件），
// 但多线程/多进程场景需要连接管理

class SQLitePool {
    constructor(dbPath, { maxConnections = 5 } = {}) {
        this.dbPath = dbPath;
        this.maxConnections = maxConnections;
        this.connections = [];
        this.queue = [];
    }

    getConnection() {
        const Database = require('better-sqlite3');
        const conn = new Database(this.dbPath);
        conn.pragma('journal_mode = WAL');
        conn.pragma('busy_timeout = 5000');
        return conn;
    }

    // 获取连接（简化版，实际生产需要更复杂的排队逻辑）
    acquire() {
        if (this.connections.length < this.maxConnections) {
            const conn = this.getConnection();
            this.connections.push(conn);
            return conn;
        }
        // 复用已有连接
        return this.connections[0];
    }

    query(sql, params = []) {
        const conn = this.acquire();
        try {
            if (sql.trim().toLowerCase().startsWith('select')) {
                return conn.prepare(sql).all(...params);
            }
            return conn.prepare(sql).run(...params);
        } finally {
            // SQLite 连接不需要显式释放（单连接复用）
        }
    }

    transaction(fn) {
        const conn = this.acquire();
        return conn.transaction(fn)();
    }

    close() {
        for (const conn of this.connections) {
            conn.close();
        }
        this.connections = [];
    }
}

// 使用
const pool = new SQLitePool('app.db');
const users = pool.query('SELECT * FROM users WHERE age > ?', [18]);
pool.close();
```
