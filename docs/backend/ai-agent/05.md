# 部署与运维

本章覆盖 AI Agent 应用从开发到生产环境的完整运维链路。

## 一、服务器要求

### 最小配置（开发/测试）

| 组件 | 配置 | 说明 |
|------|------|------|
| CPU | 2 核 | Agent 逻辑本身 CPU 消耗不高 |
| 内存 | 4 GB | Node.js + 基础服务 |
| 磁盘 | 20 GB SSD | 代码、日志、向量数据 |
| 带宽 | 5 Mbps | API 调用为主 |
| 操作系统 | Ubuntu 22.04 LTS | 推荐 |

### 生产配置

| 组件 | 配置 | 说明 |
|------|------|------|
| CPU | 4 核+ | 并发请求处理 |
| 内存 | 8 GB+ | PGVector + Redis + Node.js |
| 磁盘 | 100 GB SSD | 日志轮转、知识库备份 |
| 带宽 | 10 Mbps+ | 向量传输、文件下载 |
| 数据库 | PostgreSQL 15+ with pgvector | 向量存储 |
| 缓存 | Redis 7+ | 会话记忆、分布式锁 |

### 模型调用成本估算（DeepSeek）

| 场景 | 日均请求 | Token 消耗 | 月费用估算 |
|------|----------|-----------|-----------|
| 内部客服机器人 | 1000 次 | 50M | ¥200-500 |
| 自动化日报 Agent | 1 次 | 100K | ¥10-30 |
| 多 Agent 协作系统 | 5000 次 | 200M | ¥800-2000 |

> 实际费用取决于模型选择（DeepSeek-V3 比 R1 便宜），建议开启用量监控告警。

## 二、Docker 部署

### Dockerfile

```dockerfile
# Dockerfile
FROM node:20-alpine AS builder

WORKDIR /app
COPY package.json pnpm-lock.yaml ./
RUN npm install -g pnpm && pnpm install --frozen-lockfile

COPY . .
RUN pnpm build

# 生产镜像
FROM node:20-alpine

WORKDIR /app
ENV NODE_ENV=production

RUN npm install -g pnpm

COPY package.json pnpm-lock.yaml ./
RUN pnpm install --prod --frozen-lockfile

COPY --from=builder /app/dist ./dist
COPY --from=builder /app/data ./data

EXPOSE 3000

USER node

CMD ["node", "dist/index.js"]
```

### LLM 调用日志（关键）

```typescript
// 记录每次 LLM 调用的输入输出、Token 消耗、耗时
interface LLMCallLog {
  requestId: string
  timestamp: string
  model: string
  messages: Array<{ role: string; content: string }>
  response: string
  tokenUsage: {
    prompt: number
    completion: number
    total: number
  }
  latencyMs: number
  success: boolean
  error?: string
}

// 在 Chain/Agent 调用前后埋点
export async function withLLMLogging<T>(
  requestId: string,
  fn: () => Promise<T>,
  modelName: string
): Promise<T> {
  const start = Date.now()
  try {
    const result = await fn()
    logger.info('llm_call', {
      requestId,
      model: modelName,
      latencyMs: Date.now() - start,
      success: true,
    })
    return result
  } catch (error) {
    logger.error('llm_call_failed', {
      requestId,
      model: modelName,
      latencyMs: Date.now() - start,
      success: false,
      error: error instanceof Error ? error.message : String(error),
    })
    throw error
  }
}
```

## 四、线上问题排查

### 常见问题速查表

| 现象 | 可能原因 | 排查步骤 |
|------|----------|----------|
| 响应超时 | LLM API 慢 / 网络问题 | 查看 `llm_call` 日志中的 `latencyMs` |
| 回答质量差 | Prompt 问题 / 知识库不匹配 | 对比原始输入和 RAG 检索结果 |
| Token 费用暴增 | 循环调用 / 上下文过长 | 检查 `tokenUsage.total` 趋势 |
| 向量检索不准 | Embedding 模型不匹配 | 验证相似度分数和 Top-K 结果 |
| 对话丢失上下文 | Redis 过期 / Session 错误 | 检查 `chat:{sessionId}` 的 Key |
| Agent 死循环 | 工具调用循环 / 条件判断错误 | 开启 DEBUG 级别日志追踪步骤 |

### 排查流程

```
1. 确认服务状态
   └─ curl http://localhost:3000/health
   └─ docker-compose ps

2. 查看最近错误
   └─ docker logs --since 10m ai-agent | grep ERROR

3. 定位具体请求
   └─ 通过 requestId 串联所有日志
   └─ 检查该请求的完整调用链

4. 复现问题
   └─ 使用相同的输入在本地复现
   └─ 开启 DEBUG 日志：LOG_LEVEL=debug

5. 根因分析
   └─ LLM 层：API 响应、Token 消耗
   └─ 检索层：向量相似度、召回结果
   └─ 业务层：工具调用、状态转换
```

### 健康检查端点

```typescript
// src/api/health.ts
import { Router } from 'express'
import { Pool } from 'pg'
import Redis from 'ioredis'

export function createHealthRouter(pg: Pool, redis: Redis) {
  const router = Router()

  router.get('/health', async (req, res) => {
    const checks: Record<string, { status: string; latency?: number; error?: string }> = {}

    // 检查数据库
    const dbStart = Date.now()
    try {
      await pg.query('SELECT 1')
      checks.database = { status: 'ok', latency: Date.now() - dbStart }
    } catch (e: any) {
      checks.database = { status: 'error', error: e.message }
    }

    // 检查 Redis
    const redisStart = Date.now()
    try {
      await redis.ping()
      checks.redis = { status: 'ok', latency: Date.now() - redisStart }
    } catch (e: any) {
      checks.redis = { status: 'error', error: e.message }
    }

    // 检查 LLM API（轻量请求）
    const llmStart = Date.now()
    try {
      const response = await fetch(process.env.DEEPSEEK_BASE_URL + '/models', {
        headers: { Authorization: `Bearer ${process.env.DEEPSEEK_API_KEY}` },
      })
      checks.llm = {
        status: response.ok ? 'ok' : 'error',
        latency: Date.now() - llmStart,
      }
    } catch (e: any) {
      checks.llm = { status: 'error', error: e.message }
    }

    const allOk = Object.values(checks).every((c) => c.status === 'ok')

    res.status(allOk ? 200 : 503).json({
      status: allOk ? 'healthy' : 'unhealthy',
      timestamp: new Date().toISOString(),
      checks,
    })
  })

  return router
}
```

### 监控告警

```typescript
// 基于日志的告警规则（可对接 Prometheus/Grafana）
const alertRules = {
  // LLM API 连续失败
  llmFailure: {
    condition: (logs) =>
      logs.filter((l) => l.message === 'llm_call_failed').length > 5,
    window: '5m',
    severity: 'critical',
  },

  // Token 消耗突增
  tokenSpike: {
    condition: (logs) => {
      const recent = logs.filter((l) => l.tokenUsage?.total > 50000)
      return recent.length > 3
    },
    window: '10m',
    severity: 'warning',
  },

  // 平均延迟过高
  highLatency: {
    condition: (logs) => {
      const llmCalls = logs.filter((l) => l.latencyMs)
      const avg = llmCalls.reduce((s, l) => s + l.latencyMs, 0) / llmCalls.length
      return avg > 30000 // 30s
    },
    window: '5m',
    severity: 'warning',
  },
}
```

### 4. 缓存策略

```typescript
// 常见问题缓存
async function getCachedOrGenerate(
  question: string,
  generator: () => Promise<string>
) {
  const cacheKey = `cache:answer:${createHash('sha256').update(question).digest('hex')}`

  const cached = await redis.get(cacheKey)
  if (cached) return cached

  const answer = await generator()

  // 缓存 1 小时
  await redis.setex(cacheKey, 3600, answer)

  return answer
}
```
