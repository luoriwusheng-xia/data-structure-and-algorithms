## 一、为什么还需要 Canvas

在 DOM、CSS、SVG 已经非常强大的今天，Canvas 仍然是不可替代的渲染技术，尤其是在以下场景：

| 场景 | DOM/SVG 的问题 | Canvas 的优势 |
|------|---------------|--------------|
| 粒子特效、烟花、雨雪 | 大量 DOM 节点导致重排 | 单一画布，GPU 加速绘制 |
| 实时数据曲线（秒级/毫秒级） | DOM 更新开销大 | 直接操作像素，渲染管线短 |
| 电子签名、自由涂鸦 | SVG path 数据量大 | 位图直接绘制，导出方便 |
| 图片滤镜、视频处理 | CSS filter 能力有限 | 可逐像素处理，效果可控 |
| 2D 小游戏、物理模拟 | 碰撞检测与 DOM 不友好 | 完全掌控坐标与渲染帧 |

**Canvas 的本质是：在一张位图上，通过 JavaScript 直接控制每个像素的渲染过程。**

## 二、核心概念速览

### 2.1 高清屏适配：devicePixelRatio

普通 Canvas 在 Retina / 高分屏上会出现模糊，因为 CSS 像素与物理像素不一致。标准做法：

```javascript
const canvas = document.getElementById('canvas')
const ctx = canvas.getContext('2d')
const dpr = window.devicePixelRatio || 1

// 以 CSS 尺寸为基准，按 DPR 放大内部像素
const rect = canvas.getBoundingClientRect()
canvas.width = rect.width * dpr
canvas.height = rect.height * dpr

// 后续所有绘制逻辑都按 CSS 坐标系即可
ctx.scale(dpr, dpr)
```

### 2.2 绘制状态栈：save / restore

Canvas 的 `ctx` 是全局状态机，变换、颜色、裁剪等状态会相互影响。复杂绘制中应使用 `save()` 和 `restore()` 隔离状态。

```javascript
ctx.save()
ctx.translate(x, y)
ctx.rotate(angle)
ctx.fillStyle = 'red'
ctx.fillRect(0, 0, 100, 100)
ctx.restore() // 恢复到之前的状态
```

### 2.3 动画循环：requestAnimationFrame

永远不要用 `setInterval` 驱动 Canvas 动画。`requestAnimationFrame` 与屏幕刷新同步，且会在页面不可见时自动暂停。

```javascript
function loop() {
  // 更新逻辑
  update()
  // 绘制
  draw()
  requestAnimationFrame(loop)
}
loop()
```

### 2.4 事件坐标转换

Canvas 内部的逻辑坐标需要把鼠标/触摸坐标减去画布位置，并乘以 DPR：

```javascript
function getCanvasPoint(e, canvas, dpr) {
  const rect = canvas.getBoundingClientRect()
  return {
    x: (e.clientX - rect.left) * dpr,
    y: (e.clientY - rect.top) * dpr
  }
}
```

## 三、实战场景与案例

### 3.1 场景一：粒子特效与动画系统

**业务场景**：数据大屏 loading、节日活动页烟花、按钮点击反馈、地图飞线尾迹。

**核心思路**：

- 每个粒子是一个独立对象，维护位置、速度、生命周期。
- 在动画循环中先更新所有粒子状态，再统一绘制。
- 使用半透明背景覆盖形成拖尾残影，避免每帧全量清空。

**在线案例**：点击画布触发烟花爆炸。

**业务场景**：股票行情、IoT 监控、交易大盘、服务器指标曲线。

**业务场景**：合同签署、工单确认、手写输入、白板批注。

**核心思路**：

## 四、性能优化

Canvas 动画最容易出现的性能瓶颈是“每帧全量重绘 + 频繁创建对象 + 主线程阻塞”。下面把四种常用优化手段拆开单独说明。

### 4.1 合成模式（globalCompositeOperation）

Canvas 2D 支持 `globalCompositeOperation` 属性，它决定新绘制像素与已有像素如何混合。默认值是 `source-over`（新图形覆盖在旧图形之上）。合理利用合成模式，可以用极少的绘制调用实现擦除、水印、高亮、发光等效果。

常用模式：

| 模式 | 效果 |
|------|------|
| `source-over` | 默认，新图形覆盖旧图形 |
| `destination-out` | 新图形所在区域会“擦除”旧像素，常用于橡皮擦 |
| `lighter` / `screen` | 颜色相加/滤色，适合粒子发光、火焰 |
| `multiply` | 正片叠底，适合阴影、暗角 |
| `overlay` | 叠加，增强对比，适合做图片调色 |
| `xor` | 相交区域透明，适合特殊遮罩 |

```javascript
// 橡皮擦：用 destination-out 把已有像素清成透明
ctx.globalCompositeOperation = 'destination-out'
ctx.beginPath()
ctx.arc(x, y, 20, 0, Math.PI * 2)
ctx.fill()

// 绘制完成后恢复默认，否则会影响后续绘制
ctx.globalCompositeOperation = 'source-over'
```

```javascript
// 粒子发光：用 lighter 让重叠粒子变亮
ctx.globalCompositeOperation = 'lighter'
particles.forEach(p => {
  ctx.beginPath()
  ctx.fillStyle = `rgba(255, 100, 50, ${p.alpha})`
  ctx.arc(p.x, p.y, p.radius, 0, Math.PI * 2)
  ctx.fill()
})
ctx.globalCompositeOperation = 'source-over'
```

**注意**：合成模式是 `ctx` 全局状态，用完要立即恢复；部分模式在低版本浏览器或特定颜色格式下表现不一致，需要做降级兜底。

### 4.2 脏矩形渲染（Dirty Rectangle）

如果画面只有一小部分发生变化，每帧都 `clearRect(0, 0, width, height)` 会非常浪费。脏矩形优化的核心就是：**只清空并重绘变化区域**。

适用场景：地图上的标记点移动、数据曲线追加、白板局部笔触、游戏中少量角色移动。

不适用场景：全屏粒子、全屏滤镜、整个画面都在变化的特效。

```javascript
// 记录上一帧和当前帧的位置，取并集作为脏矩形
function getDirtyRect(prev, curr, padding = 10) {
  const minX = Math.min(prev.x, curr.x) - padding
  const minY = Math.min(prev.y, curr.y) - padding
  const maxX = Math.max(prev.x, curr.y) + padding
  const maxY = Math.max(prev.y, curr.y) + padding
  return {
    x: minX,
    y: minY,
    width: maxX - minX,
    height: maxY - minY
  }
}

function render() {
  // 只清除脏矩形
  ctx.clearRect(dirty.x, dirty.y, dirty.width, dirty.height)

  // 只重绘与脏矩形相交的对象
  objects.forEach(obj => {
    if (rectIntersect(dirty, obj.bounds)) {
      obj.draw(ctx)
    }
  })
}
```

如果有静态背景，脏矩形区域还需要先把背景贴回来，否则会出现“镂空”：

```javascript
// 假设 backgroundCanvas 是预先绘制好的静态背景
ctx.drawImage(
  backgroundCanvas,
  dirty.x, dirty.y, dirty.width, dirty.height, // 源区域
  dirty.x, dirty.y, dirty.width, dirty.height  // 目标区域
)
// 再在上面重绘动态对象
```

### 4.3 对象池（Object Pool）

粒子、飞线、标记点、弹幕等对象通常高频创建和销毁。反复 `new` 会频繁触发 GC，导致帧率抖动。对象池的思路是：**预分配一批对象，用的时候取出来，不用的时候还回去并重置状态**。

```javascript
class ParticlePool {
  constructor(createFn, resetFn, initialSize = 100) {
    this.createFn = createFn
    this.resetFn = resetFn
    this.pool = Array.from({ length: initialSize }, createFn)
    this.active = new Set()
  }

  acquire(data) {
    const item = this.pool.pop() || this.createFn()
    this.resetFn(item, data)
    this.active.add(item)
    return item
  }

  release(item) {
    if (this.active.delete(item)) {
      // 清空引用，避免内存泄漏
      this.resetFn(item, null)
      this.pool.push(item)
    }
  }

  releaseAll() {
    this.active.forEach(item => this.release(item))
  }
}
```

使用示例：

```javascript
const pool = new ParticlePool(
  () => ({ x: 0, y: 0, vx: 0, vy: 0, life: 0, maxLife: 0 }),
  (p, data) => {
    if (!data) {
      p.x = p.y = p.vx = p.vy = p.life = p.maxLife = 0
      return
    }
    p.x = data.x
    p.y = data.y
    p.vx = data.vx
    p.vy = data.vy
    p.life = 0
    p.maxLife = data.maxLife
  }
)

// 爆炸时从池中取粒子
for (let i = 0; i < 50; i++) {
  const p = pool.acquire({ x, y, vx: Math.random() * 4 - 2, vy: Math.random() * 4 - 2, maxLife: 60 })
  activeParticles.push(p)
}

// 粒子死亡时归还
activeParticles = activeParticles.filter(p => {
  p.life++
  if (p.life >= p.maxLife) {
    pool.release(p)
    return false
  }
  return true
})
```

**对象池的副作用**：池子太大会长期占用内存；对象复用时一定要彻底重置状态，否则会出现“上一个粒子残留属性”的 bug。可以设置一个上限，当池中对象超过阈值时直接丢弃，避免无限膨胀。

### 4.4 离屏渲染（Offscreen Canvas）

复杂静态图形（网格背景、地图底图、复杂图表坐标轴）如果每帧都在主画布重新绘制，会浪费大量 CPU。可以预先在离屏 Canvas 上画一次，之后每帧用 `drawImage` 贴到主画布。

```javascript
const offscreen = document.createElement('canvas')
offscreen.width = 800
offscreen.height = 400
const offCtx = offscreen.getContext('2d')

// 只执行一次：绘制复杂背景
function renderComplexBackground(ctx) {
  // 网格线、坐标轴、图例等不会频繁变化的内容
  ctx.strokeStyle = '#eee'
  for (let i = 0; i <= 800; i += 40) {
    ctx.beginPath()
    ctx.moveTo(i, 0)
    ctx.lineTo(i, 400)
    ctx.stroke()
  }
}
renderComplexBackground(offCtx)

// 主循环只贴图 + 绘制动态层
function render() {
  ctx.clearRect(0, 0, canvas.width, canvas.height)
  ctx.drawImage(offscreen, 0, 0)
  renderDynamicLayer(ctx)
  requestAnimationFrame(render)
}
```

如果静态内容非常大，可以按瓦片切分成多个离屏 Canvas，只绘制视野内的部分。

```javascript
const tiles = []
for (let row = 0; row < rows; row++) {
  for (let col = 0; col < cols; col++) {
    const tile = document.createElement('canvas')
    tile.width = TILE_SIZE
    tile.height = TILE_SIZE
    renderTile(tile.getContext('2d'), row, col)
    tiles.push({ x: col * TILE_SIZE, y: row * TILE_SIZE, canvas: tile })
  }
}

function render() {
  visibleTiles.forEach(tile => {
    ctx.drawImage(tile.canvas, tile.x, tile.y)
  })
}
```

在支持的环境中，还可以把 `OffscreenCanvas` 传给 Web Worker，把重计算放到后台线程（关于 Web Worker 的完整用法，可参考 [Web Worker 深度实战](/docs/frontend/web-worker/01)）：

```javascript
// main.js
const canvas = document.getElementById('canvas')
const offscreen = canvas.transferControlToOffscreen()
const worker = new Worker('./worker.js')
worker.postMessage({ canvas: offscreen }, [offscreen])
```

```javascript
// worker.js
self.onmessage = function (event) {
  const canvas = event.data.canvas
  const ctx = canvas.getContext('2d')
  // 在 worker 中执行重绘制，不阻塞主线程
  heavyRender(ctx)
}
```

**注意**：`transferControlToOffscreen` 后主线程不能再操作该 Canvas；Web Worker 中不支持所有 Canvas API（如部分合成模式、文字测量）。离屏 Canvas 也会占用额外内存，需要在组件卸载时释放引用。

### 4.5 控制画布分辨率

高 DPR 屏幕不要把 `canvas.width` 设置为 CSS 宽度的 3 倍甚至更高。通常限制 `dpr <= 2` 是性能与清晰度的平衡点。

```javascript
const dpr = Math.min(window.devicePixelRatio || 1, 2)
```

## 五、常见坑与最佳实践

### 5.1 坐标系混淆

Canvas 有两个坐标系：

- **CSS 尺寸**：`canvas.style.width / height`，决定页面占位。
- **逻辑尺寸**：`canvas.width / height`，决定绘制缓冲区大小。

绘制代码应按 CSS 尺寸编写，通过 `ctx.scale(dpr, dpr)` 完成缩放。

### 5.2 动画没有清理

组件卸载时必须取消 `requestAnimationFrame`，否则会造成内存泄漏和空指针异常。

```javascript
let animationId
onUnmounted(() => {
  cancelAnimationFrame(animationId)
})
```

### 5.3 触摸事件未阻止默认行为

在移动端的 Canvas 上绘图时，需要阻止 `touchmove` 的默认滚动行为。

```javascript
canvas.addEventListener('touchmove', draw, { passive: false })
```

### 5.4 图片跨域

使用 `getImageData` 处理网络图片时，图片必须满足 CORS，否则浏览器会抛出安全错误。

```javascript
const img = new Image()
img.crossOrigin = 'anonymous'
img.src = url
```

### 5.5 高频数据更新要节流

实时曲线、粒子系统等场景，数据推送频率可能高于屏幕刷新率（60Hz）。应做节流或合并：

```javascript
function throttleRender(fn, wait = 16) {
  let ticking = false
  return function (...args) {
    if (ticking) return
    ticking = true
    requestAnimationFrame(() => {
      fn.apply(this, args)
      ticking = false
    })
  }
}
```

## 总结

Canvas 是前端进阶路上必须掌握的一把利器：

- **想做好看**：掌握粒子系统、贝塞尔曲线、渐变与合成模式。
- **想做好用**：处理好 DPR、事件坐标、触摸交互与图片导出。
- **想做稳定**：管理好 `requestAnimationFrame` 生命周期，避免内存泄漏。
- **想做更快**：善用合成模式、脏矩形、对象池、离屏渲染与 Web Worker。

> 当你发现 DOM / SVG 已经无法满足性能或效果需求时，Canvas 往往就是答案。
