# WebGL 面试题精选

> 高频 WebGL 面试题及详细解答，涵盖原理、性能优化、调试技巧。

---

## 一、基础概念

### Q1：WebGL 1.0 和 WebGL 2.0 的主要区别是什么？

**答：**

| 特性 | WebGL 1.0 | WebGL 2.0 |
|------|-----------|-----------|
| 基于标准 | OpenGL ES 2.0 | OpenGL ES 3.0 |
| GLSL 版本 | `#version 100` | `#version 300 es` |
| 语法差异 | `attribute`/`varying` | `in`/`out` |
| 片元输出 | `gl_FragColor` | 自定义 `out` 变量 |
| 纹理 | 无 | 3D 纹理、2D 纹理数组 |
| MRT | 无 | 多渲染目标 (`gl.drawBuffers`) |
| VAO | 可选扩展 | 原生支持 |
| 实例渲染 | 扩展 | 原生支持 |
| 查询对象 | 无 | ` occlusion query` |
| 变换反馈 | 扩展 | 原生支持 |

---

### Q2：WebGL 渲染管线有哪些阶段？

**答：**

```
顶点数据 → 顶点着色器 → 细分着色器(可选) → 几何着色器(可选)
  → 图元装配 → 光栅化 → 片元着色器
  → 逐片元操作(深度测试、混合等) → 帧缓冲
```

关键阶段说明：

1. **顶点着色器**：处理每个顶点，输出裁剪空间坐标
2. **图元装配**：将顶点组装为点/线/三角形
3. **光栅化**：将图元离散化为像素（片元）
4. **片元着色器**：计算每个片元的颜色
5. **逐片元操作**：深度测试、模板测试、混合、裁剪

---

### Q3：什么是裁剪空间（Clip Space）？NDC 是什么？

**答：**

- **裁剪空间**：顶点着色器输出的坐标空间，`xyzw` 四维向量
- **NDC（Normalized Device Coordinates）**：执行透视除法 `x/w, y/w, z/w` 后的空间
  - NDC 范围：`x,y` ∈ [-1, 1]，`z` ∈ [-1, 1]（WebGL）或 [0, 1]（Vulkan/DirectX）
- **屏幕空间**：NDC 经视口变换后映射到像素坐标

## 二、着色器与 GLSL

### Q4：uniform、attribute/varying 的区别？

**答：**

| 类型 | 作用域 | 可变性 | 用途 |
|------|--------|--------|------|
| `attribute` / `in` | 仅顶点着色器 | 每顶点不同 | 顶点位置、颜色、法线、UV |
| `uniform` | 两着色器共用 | 绘制调用期间不变 | MVP矩阵、光照参数、纹理采样器 |
| `varying` / `out-in` | 顶点→片元 | 经光栅化插值 | 颜色、UV、法线在片元间插值 |

---

### Q5：什么是 `precision mediump float`？为什么需要它？

**答：**

GLSL ES 要求片元着色器显式声明浮点精度，有三种级别：

- `highp`：高精度（16-32位），计算量大，部分移动设备不支持
- `mediump`：中等精度（10-16位），大多数情况够用，性能平衡
- `lowp`：低精度（8-10位），用于颜色等不需要精度的场景

顶点着色器默认 `highp`，片元着色器无默认精度必须显式声明。

---

### Q6：如何避免深度冲突（Z-fighting）？

**答：**

深度冲突发生在两个面非常接近时，深度缓冲精度不足以区分：

1. **增加近平面距离**：`near` 不要太小（如用 0.1 而非 0.001）
2. **使用对数深度缓冲**：非线性深度分配提高远处精度
3. **多边形偏移**：`gl.enable(gl.POLYGON_OFFSET_FILL)` + `gl.polygonOffset`
4. **避免共面**：建模时保留微小间距

## 三、性能优化

### Q7：如何减少 WebGL 的 draw call？

**答：**

| 技术 | 原理 | 适用场景 |
|------|------|----------|
| **合批（Batching）** | 合并顶点缓冲，一次 draw 多个物体 | 静态场景、同类材质 |
| **实例渲染（Instancing）** | `ANGLE_instanced_arrays` 一次绘制多个实例 | 大量相同几何（草、树、粒子） |
| **纹理图集（Atlas）** | 多张小图合并为一张大图 | UI、2D 精灵、多样纹理 |
| **细节层次（LOD）** | 远距离使用低模 | 大场景、开放世界 |
| **视锥剔除（Frustum Culling）** | 只渲染视锥内的物体 | 任意 3D 场景 |
| **遮挡查询（Occlusion Query）** | GPU 查询物体是否被遮挡 | WebGL 2.0 复杂场景 |

---

### Q8：如何优化纹理内存和带宽？

**答：**

1. **压缩纹理**：使用 GPU 原生格式（DXT/ETC/ASTC），减少显存占用
2. **Mipmap**：生成多级纹理，远距采样低分辨率层级，减少缓存未命中
3. **纹理尺寸**：保持为 2 的幂次方（如 512x512、1024x1024）
4. **NPOT 纹理**：非 2 幂纹理在 WebGL 1.0 中有限制（无法 mipmap、repeat）
5. **纹理复用**：多个材质共用同一张纹理

---

### Q9：WebGL 中 alpha 混合的常见问题？

**答：**

**问题 1：透明物体渲染顺序**

半透明物体必须从远到近绘制，否则深度测试会导致近处半透明面遮挡远处面。

**问题 2：混合模式**

```javascript
gl.enable(gl.BLEND);
gl.blendFunc(gl.SRC_ALPHA, gl.ONE_MINUS_SRC_ALPHA); // 标准混合
gl.blendFunc(gl.SRC_ALPHA, gl.ONE);                 // 加法混合（光效）
```

**问题 3：深度写入**

半透明物体通常关闭深度写入（`gl.depthMask(false)`），但保留深度测试，防止被不透明物体遮挡。

## 四、高级话题

### Q10：什么是延迟渲染（Deferred Rendering）？优缺点？

**答：**

**原理**：不直接计算光照，先渲染到 G-Buffer（存储位置、法线、颜色、材质），再对 G-Buffer 进行光照计算。

**优点**：
- 光源数量与场景复杂度解耦，千盏光源无压力
- 每像素仅计算一次光照

**缺点**：
- 内存开销大（G-Buffer 多目标渲染）
- 不支持透明物体（需前向渲染补充）
- 全屏光照pass有带宽压力
- 抗锯齿复杂（不能直接用 MSAA）

---

### Q11：PBR（基于物理的渲染）核心概念？

**答：**

PBR 遵循能量守恒和微表面模型：

1. **金属度/粗糙度工作流**：
   - `baseColor`：绝缘体为漫反射色，金属为反射率
   - `metallic`：0=绝缘体，1=金属
   - `roughness`：表面粗糙程度，影响反射模糊

2. **BRDF（双向反射分布函数）**：
   - 漫反射项：Lambert 或更复杂的模型
   - 镜面反射项：Cook-Torrance 微表面模型

3. **IBL（基于图像的光照）**：
   - 使用环境贴图（HDR）提供间接光照
   - 需预计算 irradiance map 和 prefiltered map

### Q12：WebGL 中如何实现物体拾取（Picking）？

**答：**

**方法 1：颜色编码（GPU Picking）**

为每个物体分配唯一 ID 颜色，渲染到离屏 FBO，读取鼠标位置像素颜色反查物体。

```javascript
// 渲染时
const idColor = new Float32Array([
  ((id & 0xff0000) >> 16) / 255,
  ((id & 0x00ff00) >> 8) / 255,
  (id & 0x0000ff) / 255,
  1.0
]);
gl.uniform4fv(uIdColorLoc, idColor);

// 读取像素
const pixel = new Uint8Array(4);
gl.readPixels(mouseX, canvasHeight - mouseY, 1, 1, gl.RGBA, gl.UNSIGNED_BYTE, pixel);
const pickedId = (pixel[0] << 16) | (pixel[1] << 8) | pixel[2];
```

**方法 2：射线检测（Ray Casting）**

从相机发射射线，与物体包围盒或三角形求交。CPU 计算，适合简单场景。

## 五、调试与排错

### Q13：WebGL 程序编译/链接失败的排查方法？

**答：**

```javascript
// 检查编译错误
if (!gl.getShaderParameter(shader, gl.COMPILE_STATUS)) {
  console.error('Shader compile error:', gl.getShaderInfoLog(shader));
}

// 检查链接错误
if (!gl.getProgramParameter(program, gl.LINK_STATUS)) {
  console.error('Program link error:', gl.getProgramInfoLog(program));
}

// 检查 active attributes/uniforms
const numAttribs = gl.getProgramParameter(program, gl.ACTIVE_ATTRIBUTES);
for (let i = 0; i < numAttribs; i++) {
  console.log(gl.getActiveAttrib(program, i));
}
```

### Q14：WebGL 常见的性能瓶颈及排查？

**答：**

使用 `Spector.js` 浏览器插件或 Chrome GPU 性能面板分析：

| 瓶颈类型 | 表现 | 解决方案 |
|----------|------|----------|
| CPU 端瓶颈 | JavaScript 耗时高 | 减少 JS 计算，使用 Web Worker |
| Draw Call 过多 | 大量小批次绘制 | 合批、实例化渲染 |
| 顶点处理瓶颈 | 顶点着色器复杂 | 简化顶点着色器，使用 LOD |
| 片元处理瓶颈 | 分辨率敏感 | 简化片元着色器，降低分辨率 |
| 带宽瓶颈 | 大量纹理/顶点上传 | 压缩纹理、减少顶点属性 |
| 状态切换 | 频繁绑定纹理/缓冲 | 排序渲染，减少状态变更 |

## 六、与其他技术对比

### Q15：WebGL、WebGPU、Three.js 的关系？

**答：**

| 技术 | 层级 | 特点 |
|------|------|------|
| **WebGL** | 底层 GPU API | 基于 OpenGL ES，成熟但 API 老旧 |
| **WebGPU** | 底层 GPU API | 基于 Vulkan/Metal/D3D12，现代设计，计算着色器 |
| **Three.js** | 高层框架 | 封装 WebGL/WebGPU，提供场景图、材质、光照、加载器 |

关系：Three.js 可以运行在 WebGL 和 WebGPU 渲染器之上，WebGPU 是 WebGL 的下一代继任者。

---

### Q16：什么场景下应该直接使用 WebGL 而不是 Three.js？

**答：**

直接使用 WebGL 的场景：

1. **极致性能优化**：需要完全控制渲染流程，如粒子系统、大规模数据可视化
2. **特殊渲染需求**：自定义渲染管线（延迟渲染、体渲染）、后处理链
3. **极小体积**： Three.js 即使 tree-shaking 也有几十 KB，纯 WebGL 可更小
4. **学习目的**：理解底层图形原理
5. **2D 渲染**：大量 2D 精灵，使用纯 WebGL 可能比 Three.js 更轻量

使用 Three.js 的场景：常规 3D 应用、需要快速开发、需要丰富生态（加载器、控制器、后期处理）。
