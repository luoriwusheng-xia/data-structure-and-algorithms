# 插件架构设计

插件（Plugin）是现代前端工具链的"黏合剂"。无论是 Rollup、Vite、webpack 这类构建工具，还是 Vue、Express 这类应用框架，插件机制都让核心保持精简，同时允许开发者以声明式的方式扩展能力。理解这些设计思路，能帮助我们写出更通用、可维护的工具和框架。

## 1. 插件架构要解决什么问题

一个成熟的插件系统通常要回答以下几个问题：

- **扩展点在哪里**：在哪些生命周期节点上允许外部代码介入？
- **如何组合**：多个插件之间是并行、串行还是竞争关系？
- **执行顺序怎么控制**：如何决定谁先谁后？
- **上下文与状态如何共享**：插件之间如何安全地交换信息？
- **性能如何保障**：如何避免无意义的插件调用？

不同工具对上述问题的回答方式不同，但本质上都围绕 **"钩子（Hook）+ 上下文（Context）"** 这两个核心抽象展开。

## 2. 核心抽象：Hook、Context 与 Pipeline

### 2.1 Hook

Hook 是框架暴露给插件的"挂载点"。插件通过注册 Hook 来参与框架内部流程。根据执行语义，Hook 通常分为几类：

| 类型 | 语义 | 典型场景 |
| --- | --- | --- |
| `parallel` | 并行执行，互不阻塞 | 初始化、清理 |
| `sequential` | 串行执行，等待前一个完成 | 代码转换、资源处理 |
| `first` | 找到第一个返回非空值即停止 | 模块解析、ID 查找 |
| `waterfall` | 结果依次传递，后一个拿到前一个的输出 | 配置合并、数据转换 |
| `bail` | 任一返回非空即中断 | 校验、短路 |

### 2.2 Context

Context 是插件在生命周期中可以访问的框架内部对象，例如 Rollup 的 `this` 上下文、webpack 的 `compiler`/`compilation`、Vite 的 `ViteDevServer`。Context 既提供服务（如日志、文件系统、模块信息），也承载状态。

### 2.3 Pipeline

Pipeline 描述插件执行的完整管线。以构建工具为例，通常包含：

```
options -> buildStart -> resolveId -> load -> transform -> moduleParsed -> buildEnd -> outputOptions -> renderStart -> renderChunk -> generateBundle -> writeBundle -> closeBundle
```

插件只需要注册自己关心的阶段，不必关心完整的流程。

## 3. 主流工具的插件设计

### 3.1 Rollup：最小而清晰的钩子模型

Rollup 的插件模型非常纯粹：一个插件是一个对象，包含 `name` 和若干 Hook。

```javascript
export default function myRollupPlugin(options = {}) {
  return {
    name: 'my-plugin',

    // 构建开始
    buildStart(inputOptions) {
      console.log('build start')
    },

    // 模块解析：first 类型，第一个返回有效值的插件获胜
    resolveId: {
      order: 'pre',
      handler(source) {
        if (source === 'virtual:foo') {
          return { id: source, external: false }
        }
        return null
      }
    },

    // 加载模块
    load(id) {
      if (id === 'virtual:foo') {
        return 'export const msg = "hello"'
      }
    },

    // 代码转换：sequential 类型
    transform(code, id) {
      if (id.endsWith('.js')) {
        return { code: code.replace(/console\.log/g, 'void') }
      }
    },

    // 构建结束
    buildEnd(error) {
      if (error) console.error(error)
    }
  }
}
```

Rollup 的 Hook 执行顺序非常清晰：

```
options -> buildStart -> resolveId -> load -> transform -> moduleParsed -> buildEnd
```

最新版本 Rollup 还引入了 `order` 和 `filter` 能力，用于精细化控制执行：

```javascript
{
  resolveId: {
    order: 'pre', // 'pre' | null | 'post'
    handler(source) { /* ... */ }
  },
  transform: {
    filter: { id: '*.jsx', code: '<MyComponent' },
    handler(code, id) { /* 只处理包含 <MyComponent 的 .jsx 文件 */ }
  }
}
```

`filter` 可以显著减少插件的无意义调用，是性能优化的重要手段。

### 3.2 Vite 8：Rollup 超集 + 开发服务器

Vite 8 的插件完全兼容 Rollup 插件，同时为了兼顾开发服务器和 HMR，扩展了自己的钩子。

```javascript
export default function myVitePlugin() {
  let server

  return {
    name: 'my-vite-plugin',

    // 执行顺序
    enforce: 'pre',

    // 只在 serve 或 build 阶段生效
    apply: 'serve',

    // 修改 Vite 配置
    config(config, { command, mode }) {
      return {
        resolve: {
          alias: { '@utils': '/src/utils' }
        }
      }
    },

    // 获取解析后的配置
    configResolved(resolvedConfig) {
      console.log('root:', resolvedConfig.root)
    },

    // 配置开发服务器
    configureServer(_server) {
      server = _server
      server.middlewares.use('/api/custom', (req, res, next) => {
        res.end(JSON.stringify({ custom: 'response' }))
      })
    },

    // 处理 HMR
    handleHotUpdate({ file, server, modules }) {
      if (file.endsWith('.custom')) {
        server.ws.send({ type: 'custom', event: 'custom-update', data: { file } })
        return []
      }
      return modules
    },

    // Rollup 兼容钩子
    resolveId(id) { /* ... */ },
    load(id) { /* ... */ },
    transform(code, id) { /* ... */ }
  }
}
```

Vite 8 的设计要点：

- **兼容 Rollup 插件**：生态可以无缝复用。
- **`enforce` 控制顺序**：`pre` 在核心插件之前执行，`post` 在之后。
- **`apply` 限制作用域**：避免插件在不合适的阶段运行。
- **服务器钩子**：`configureServer`、`configurePreviewServer`、`handleHotUpdate` 让插件能介入开发服务器。

### 3.3 webpack 5：Tapable 事件驱动

webpack 5 的插件系统基于 Tapable 库。插件不再是简单的 Hook 对象，而是一个带有 `apply(compiler)` 方法的类。

```javascript
class MyWebpackPlugin {
  apply(compiler) {
    // 同步钩子
    compiler.hooks.compile.tap('MyWebpackPlugin', (params) => {
      console.log('compile start')
    })

    // 异步串行钩子
    compiler.hooks.emit.tapAsync('MyWebpackPlugin', (compilation, callback) => {
      console.log('emit assets')
      callback()
    })

    // Promise 钩子
    compiler.hooks.run.tapPromise('MyWebpackPlugin', (compiler) => {
      return Promise.resolve()
    })

    // 嵌套钩子：访问 compilation
    compiler.hooks.compilation.tap('MyWebpackPlugin', (compilation) => {
      compilation.hooks.optimize.tap('MyWebpackPlugin', () => {
        console.log('optimize')
      })

      compilation.hooks.processAssets.tap(
        {
          name: 'MyWebpackPlugin',
          stage: compiler.webpack.Compilation.PROCESS_ASSETS_STAGE_ADDITIONS
        },
        (assets) => {
          // 处理产物
        }
      )
    })
  }
}

module.exports = MyWebpackPlugin
```

webpack 5 的 Hook 类型非常丰富：

| Tapable 类型 | 执行方式 | 示例 |
| --- | --- | --- |
| `SyncHook` | 同步，不关心返回值 | `compile` |
| `SyncBailHook` | 同步，返回非空即短路 | `resolve` |
| `SyncWaterfallHook` | 同步，结果依次传递 | 配置处理 |
| `AsyncParallelHook` | 异步并行 | `make` |
| `AsyncSeriesHook` | 异步串行 | `emit` |
| `AsyncSeriesBailHook` | 异步串行，可短路 | `beforeResolve` |

webpack 的设计特点是 **"事件驱动 + 强上下文"**。插件通过 `tap` 注册回调，框架通过 `call` 触发事件。这种方式非常灵活，但学习曲线也更陡峭。

### 3.4 Vue：应用级插件注册

Vue 的插件系统是面向应用（App）的。插件通过一个 `install` 方法扩展应用能力。

```javascript
// 对象形式
const i18nPlugin = {
  install(app, options) {
    app.config.globalProperties.$t = (key) => {
      return options.locale[key] || key
    }
    app.provide('i18n', options)
  }
}

// 函数形式
const loggerPlugin = (app, options) => {
  app.config.globalProperties.$log = (...args) => {
    if (options.enabled) console.log(...args)
  }
}

// 使用
const app = createApp(App)
app.use(i18nPlugin, { locale: { hello: '你好' } })
app.use(loggerPlugin, { enabled: true })
app.mount('#app')
```

Vue 3 的 Plugin 类型定义也很有意思：

```typescript
type PluginInstallFunction<Options = any[]> = Options extends unknown[]
  ? (app: App, ...options: Options) => any
  : (app: App, options: Options) => any

type ObjectPlugin<Options = any[]> = {
  install: PluginInstallFunction<Options>
}

type FunctionPlugin<Options = any[]> = PluginInstallFunction<Options> &
  Partial<ObjectPlugin<Options>>

type Plugin<Options = any[]> = FunctionPlugin<Options> | ObjectPlugin<Options>
```

Vue 插件设计的核心：

- **单一入口**：通过 `app.use` 统一注册。
- **选项透传**：插件可以接收任意配置。
- **能力扩展**：通过 `app.provide`、`app.component`、`app.directive`、`app.config.globalProperties` 等方式注入能力。
- **与响应式系统解耦**：插件本身不强制依赖 Vue 的响应式实现。

## 4. 从零实现一个最小插件系统

为了理解这些设计的共性，我们可以实现一个极简的插件调度器：

```javascript
class PluginSystem {
  constructor() {
    this.hooks = new Map()
    this.plugins = []
  }

  // 注册钩子类型
  registerHook(name, type = 'parallel') {
    if (!this.hooks.has(name)) {
      this.hooks.set(name, { type, handlers: [] })
    }
  }

  // 使用插件
  use(plugin, options) {
    if (typeof plugin === 'function') {
      plugin = { install: plugin }
    }
    this.plugins.push(plugin)
    plugin.install(this, options)
  }

  // 插件注册钩子处理器
  tap(hookName, handler, order = 0) {
    const hook = this.hooks.get(hookName)
    if (!hook) throw new Error(`Hook ${hookName} not found`)
    hook.handlers.push({ handler, order })
  }

  // 触发钩子
  async call(hookName, context, input) {
    const hook = this.hooks.get(hookName)
    if (!hook) return input

    const handlers = hook.handlers.slice().sort((a, b) => a.order - b.order)

    if (hook.type === 'first') {
      for (const { handler } of handlers) {
        const result = await handler(context, input)
        if (result != null) return result
      }
      return null
    }

    if (hook.type === 'waterfall') {
      let result = input
      for (const { handler } of handlers) {
        result = await handler(context, result)
      }
      return result
    }

    if (hook.type === 'sequential') {
      for (const { handler } of handlers) {
        await handler(context, input)
      }
      return
    }

    // parallel
    await Promise.all(handlers.map(({ handler }) => handler(context, input)))
  }
}

// 使用示例
const system = new PluginSystem()
system.registerHook('init', 'parallel')
system.registerHook('resolve', 'first')
system.registerHook('transform', 'waterfall')

system.use({
  install(system) {
    system.tap('resolve', (ctx, id) => {
      if (id === 'virtual:foo') return { id, content: 'export default 1' }
      return null
    }, -1)
  }
})

system.use((system) => {
  system.tap('transform', (ctx, code) => code + '\n/* transformed */')
})

;(async () => {
  await system.call('init', {}, null)
  const resolved = await system.call('resolve', {}, 'virtual:foo')
  const transformed = await system.call('transform', {}, resolved.content)
  console.log(transformed)
})()
```

这个最小实现覆盖了：

- 插件注册（`use`）
- 钩子类型（`parallel`、`first`、`waterfall`、`sequential`）
- 执行顺序控制（`order`）
- 上下文传递

## 5. 设计插件架构的最佳实践

综合 Rollup、Vite、webpack 和 Vue 的设计，可以总结出以下实践：

### 5.1 明确 Hook 的语义

不要为了灵活而滥用 Hook。每个 Hook 应该有清晰的执行语义：是并行还是串行？是否允许短路？是否返回结果？

### 5.2 提供执行顺序控制

当多个插件注册同一个 Hook 时，框架应该提供 `order`、`enforce`、`stage` 等机制，让开发者能预测执行顺序。

### 5.3 使用 filter 减少无效调用

Rollup 的 `filter` 和 Vite 的 `apply` 都是性能优化的典型手段。插件不应该在每个文件上都执行无意义的逻辑。

### 5.4 保持 Context 的稳定性

Context 对象应该是插件之间共享信息的唯一通道。避免让插件直接修改框架内部状态，而是通过 Context 提供的 API 进行交互。

### 5.5 错误处理与边界

- 在 `buildEnd`、`closeBundle` 等收尾钩子中处理错误。
- 对异步 Hook 提供超时或取消机制。
- 插件抛错时，应该能定位到具体插件名称（因此 `name` 字段很重要）。

### 5.6 类型安全

现代插件系统应该提供 TypeScript 类型定义，例如 Rollup 的 `Plugin` 类型、Vite 的 `Plugin` 类型、Vue 的 `Plugin` 类型。类型不仅能提升开发体验，也能约束插件的行为。

## 6. 总结

| 工具 | 核心模型 | 插件形态 | 特点 |
| --- | --- | --- | --- |
| Rollup | 构建管线 Hook | 对象（name + hooks） | 简洁、清晰、可预测 |
| Vite 8 | Rollup 超集 + 服务器 Hook | 对象（含 Vite 专属钩子） | 开发/生产统一，生态复用 |
| webpack 5 | Tapable 事件系统 | 类（apply 方法） | 灵活、强大、复杂 |
| Vue | 应用能力注入 | 对象/函数（install） | 面向应用，注入式扩展 |

插件架构的本质是 **"在正确的时间，把正确的上下文暴露给正确的扩展代码"**。设计一个插件系统时，不必追求像 webpack 一样复杂，也不一定要像 Rollup 一样极简，关键是根据你的工具或框架的使用场景，选择合适的 Hook 模型、执行语义和上下文边界。
