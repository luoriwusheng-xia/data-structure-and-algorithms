# 前端实时协同实战指南

---

## 一、什么是实时协同

实时协同（Real-time Collaboration）指多个用户在各自终端上同时编辑同一份文档、画布或数据，并在毫秒级时间内看到彼此的变化。

典型场景：

- 在线文档：飞书文档、Notion、腾讯文档、Google Docs
- 白板协同：Figma、Excalidraw、墨刀
- 代码协作：VS Code Live Share
- 低代码平台：多人同时拖拽组件、配置属性

协同系统需要同时解决三个问题：

1. **实时性**：本地操作立即生效，他人变更快速同步。
2. **一致性**：所有用户最终看到相同结果。
3. **可用性**：弱网、离线、并发冲突下仍能正常工作。

---

## 二、实时协同的核心问题

| 问题 | 说明 | 解决方案 |
|------|------|---------|
| 操作传输 | 本地操作如何高效广播给其他人 | WebSocket、SSE、长轮询 |
| 并发冲突 | 多人同时修改同一位置产生冲突 | OT、CRDT、锁、版本向量 |
| 状态合并 | 离线重连后如何合并本地与远端变更 | CRDT 自动合并、OT 服务端转换 |
| 感知与光标 | 展示他人位置、选区、在线状态 | 心跳、presence 通道 |
| 权限与undo | 谁能改、误操作能否撤回 | 操作权限、inverse operation |

## 三、传输方案对比

### 3.1 WebSocket

双向低延迟通道，是目前实时协同的首选。

```javascript
const ws = new WebSocket('wss://api.example.com/collab')

ws.onopen = () => {
  ws.send(JSON.stringify({
    type: 'join',
    roomId: 'doc-123',
    userId: 'u-001'
  }))
}

ws.onmessage = (event) => {
  const message = JSON.parse(event.data)
  applyRemoteOperation(message)
}

function broadcastOperation(op) {
  ws.send(JSON.stringify({
    type: 'op',
    roomId: 'doc-123',
    op
  }))
}
```

### 3.2 Server-Sent Events (SSE)

适合以服务端推送为主的单向场景，例如直播评论、状态广播。

```javascript
const source = new EventSource('/collab/doc-123/events')

source.onmessage = (event) => {
  const message = JSON.parse(event.data)
  applyRemoteOperation(message)
}

// 本地操作仍通过 HTTP POST 发送
async function sendOperation(op) {
  await fetch('/collab/doc-123/op', {
    method: 'POST',
    headers: { 'Content-Type': 'application/json' },
    body: JSON.stringify(op)
  })
}
```

### 3.3 长轮询

兼容性最好但延迟高，通常作为降级方案。

| 方案 | 延迟 | 双向通信 | 复杂度 | 推荐场景 |
|------|------|---------|--------|---------|
| WebSocket | 低 | 是 | 中 | 实时协同首选 |
| SSE | 低 | 否 | 低 | 服务端主导推送 |
| 长轮询 | 中 | 否 | 低 | 兼容性要求高的降级 |

## 四、冲突解决：OT 与 CRDT

### 4.1 Operational Transformation（OT）

OT 的核心思想：当本地操作与远端操作并发时，服务端根据顺序对操作进行变换，使其在目标状态下正确生效。

```
用户 A: 在位置 0 插入 "a"
用户 B: 在位置 0 插入 "b"

服务端按顺序处理：
  A 的操作先执行：文本变为 "a"
  B 的操作需要变换：原本在 0 插入，因 A 已在 0 插入一个字符，B 的插入位置后移 1，变为在 1 插入
  最终结果："ab"
```

OT 依赖中央服务器做变换，实现复杂，尤其在富文本、结构化文档中难以维护。

### 4.2 Conflict-free Replicated Data Type（CRDT）

CRDT 让每个客户端独立演化状态，只要满足数学性质，最终一定能自动合并一致。

常用文本 CRDT 类型：

- **LSEQ**：基于序列的分配策略，适合线性结构。
- **YATA / RGA**：基于双向链表，合并自然。
- **Logoot / WOOT**：早期文本 CRDT，逻辑位置稳定。

```javascript
// 简化示意：CRDT 每个字符携带唯一 ID
{
  id: ['u-001', 3],
  value: 'h',
  left: null,
  right: ['u-002', 5]
}
```

| 特性 | OT | CRDT |
|------|-----|------|
| 是否需要中心服务器 | 是 | 否 |
| 离线支持 | 弱 | 强 |
| 实现难度 | 高 | 中 |
| 扩展性 | 差 | 好 |
| 内存开销 | 低 | 较高 |

> 现代前端协同系统越来越多采用 CRDT，代表项目：Yjs、Automerge。

## 五、Yjs 实战：基于 CRDT 的协同编辑

### 5.1 安装与基础用法

```bash
npm install yjs y-websocket
```

```javascript
import * as Y from 'yjs'
import { WebsocketProvider } from 'y-websocket'

const ydoc = new Y.Doc()
const provider = new WebsocketProvider(
  'wss://api.example.com/yjs',
  'room-doc-123',
  ydoc
)

const ytext = ydoc.getText('content')

// 监听远端变更
ytext.observe(() => {
  console.log('当前内容：', ytext.toString())
})

// 本地插入
ytext.insert(0, 'Hello')
```

### 5.2 绑定到编辑器

```javascript
import * as Y from 'yjs'
import { MonacoBinding } from 'y-monaco'
import { WebsocketProvider } from 'y-websocket'
import * as monaco from 'monaco-editor'

const ydoc = new Y.Doc()
const provider = new WebsocketProvider('wss://...', 'room', ydoc)
const ytext = ydoc.getText('monaco')

const editor = monaco.editor.create(document.getElementById('editor'), {
  value: '',
  language: 'javascript'
})

const binding = new MonacoBinding(
  ytext,
  editor.getModel(),
  new Set([editor]),
  provider.awareness
)
```

## 六、协同状态同步模式

### 6.1 客户端优先（Optimistic）

本地操作立即应用，随后异步同步给服务端。体验最流畅，但需要处理冲突与回滚。

```javascript
function localInsert(index, text) {
  // 1. 立即更新 UI
  editor.insert(index, text)

  // 2. 生成操作并发送
  const op = { type: 'insert', index, text, version: localVersion }
  pendingOps.push(op)
  broadcastOperation(op)
}
```

### 6.2 服务端确认后应用（Pessimistic）

本地操作发送后等待服务端确认才更新 UI。实现简单但延迟明显，不适合实时编辑。

### 6.3 推荐：乐观更新 + 服务端权威

- 本地立即渲染。
- 服务端作为最终权威，返回转换后的确认或纠错。
- 若服务端拒绝或返回差异，本地做 patch 修正。

function renderRemoteSelection(user, range) {

## 八、实战：手写一个最小协同编辑器

```javascript
import * as Y from 'yjs'

class SimpleCollabEditor {
  constructor(element, roomId) {
    this.el = element
    this.ydoc = new Y.Doc()
    this.ytext = this.ydoc.getText('content')

    // 模拟网络层
    this.provider = new MockProvider(roomId, this.ydoc)

    this.el.addEventListener('input', this.handleInput.bind(this))
    this.ytext.observe(() => {
      this.updateView(false)
    })
  }

  handleInput(e) {
    const { inputType } = e
    const selectionStart = this.el.selectionStart

    if (inputType === 'insertText') {
      this.ytext.insert(selectionStart - 1, e.data)
    } else if (inputType === 'deleteContentBackward') {
      this.ytext.delete(selectionStart, 1)
    }
  }

  updateView(fromLocal = true) {
    const current = this.el.value
    const next = this.ytext.toString()

    if (current === next) return

    const start = this.el.selectionStart
    this.el.value = next

    if (fromLocal) {
      this.el.setSelectionRange(start, start)
    }
  }
}
```

## 九、生产落地 checklist

| 检查项 | 说明 |
|--------|------|
| 选择 CRDT 框架 | 推荐使用 Yjs 或 Automerge，避免自研 OT |
| 连接恢复 | WebSocket 断线后自动重连，合并离线期间的本地变更 |
| 操作幂等 | 每个操作携带唯一 ID，防止重复应用 |
| 权限控制 | 只读、评论、编辑等角色在服务端校验 |
| 版本快照 | 定期保存文档快照，加速新用户加入 |
| 性能优化 | 大文档分片、增量同步、debounce 本地输入 |
| 光标防抖 | 高频光标移动使用节流，避免网络拥塞 |
| 可撤销重做 | 基于 inverse operation 或 Yjs undoManager |

---

> 实时协同的难点不在“实时”，而在“一致”。优先选择成熟的 CRDT 方案，把精力投入到产品体验与边界场景处理上。
