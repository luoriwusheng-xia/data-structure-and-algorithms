What

什么是 WebRTC？

WebRTC（Web Real-Time Communication）是 浏览器内置的一套实时音视频通信 + P2P 数据传输技术栈。

无需安装插件，几行 JavaScript 就能实现浏览器之间的视频通话。

📹

音视频通话

浏览器自带的"微信视频通话"

📡

P2P 数据传输

浏览器之间直接传文件的"快递通道"

🖥️

屏幕共享

把浏览器标签页/整个屏幕"直播"给对方

Why

为什么需要 WebRTC？

❌ 传统方案痛点

WebSocket 中转

  • 所有数据经过服务器，延迟高
  • 服务器带宽成本爆炸
  • 并发一高就扛不住
浏览器 A → WebSocket Server → 浏览器 B

✅ WebRTC 优势

P2P 直连

  • 延迟 20~80ms（直连）
  • 服务器只传信令，零媒体流量
  • 去中心化，用户越多越省
浏览器 A ⟷ 浏览器 B
（数据不经过服务器）
维度 WebSocket 中转 WebRTC P2P
延迟 50~200ms 20~80ms
带宽成本 服务器承担全部流量 服务器只传信令
扩展性 服务器是瓶颈 去中心化

API 1 / 3

getUserMedia

获取摄像头 / 麦克风 / 屏幕共享的媒体流

// 获取摄像头 + 麦克风
async function getLocalStream() {
  const stream = await navigator.mediaDevices.getUserMedia({
    video: {
      width: { ideal: 1280 },
      height: { ideal: 720 },
      facingMode: 'user' // 'user' | 'environment'
    },
    audio: {
      echoCancellation: true,   // 回声消除
      noiseSuppression: true,   // 噪声抑制
      autoGainControl: true     // 自动增益
    }
  })

  // 绑定到 <video> 元素
  videoElement.srcObject = stream
  return stream
}

// 屏幕共享
const screenStream = await navigator.mediaDevices.getDisplayMedia({
  video: { cursor: 'always', displaySurface: 'monitor' },
  audio: true
})

API 2 / 3

RTCPeerConnection

管理点对点连接的核心 —— "智能路由器"

// 创建 PeerConnection
const pc = new RTCPeerConnection({
  iceServers: [
    { urls: 'stun:stun.l.google.com:19302' },  // STUN: 发现公网 IP
    { urls: 'turn:your-turn.com:3478',          // TURN: P2P 失败时中转
      username: 'user', credential: 'pass' }
  ]
})

// 1. 添加本地媒体轨道
localStream.getTracks().forEach(track => {
  pc.addTrack(track, localStream)
})

// 2. 监听远端轨道
pc.ontrack = (event) => {
  remoteVideo.srcObject = event.streams[0]
}

// 3. 收集并发送 ICE 候选
pc.onicecandidate = (event) => {
  if (event.candidate) {
    signalingServer.send({ type: 'ice-candidate', candidate: event.candidate })
  }
}

// 4. 监听连接状态
pc.onconnectionstatechange = () => {
  console.log('状态:', pc.connectionState)
  // new → connecting → connected → disconnected → failed → closed
}

API 3 / 3

RTCDataChannel

P2P 版本的 WebSocket —— 在连接上传输任意数据

// 创建 DataChannel（在已有 PeerConnection 上）
const dataChannel = pc.createDataChannel('chat', {
  ordered: true,        // 保证消息按序到达
  maxRetransmits: 3     // 最大重传次数
})

dataChannel.onopen = () => {
  console.log('通道已连接')
  dataChannel.send('Hello WebRTC!')
}

dataChannel.onmessage = (event) => {
  console.log('收到:', event.data)
}

// 接收方通过事件获取
pc.ondatachannel = (event) => {
  const channel = event.channel
  channel.onmessage = (e) => console.log('收到:', e.data)
}

可靠模式（类似 TCP）

ordered: true, maxRetransmits: null

适合：文件传输、文字聊天

不可靠模式（类似 UDP）

ordered: false, maxRetransmits: 0

适合：实时游戏状态同步

Signaling

信令流程

WebRTC 本身不规定信令协议，但必须先交换元数据才能建立连接

浏览器 A
发起方
SDP Offer
ICE Candidate
信令服务器
Socket.IO
SDP Answer
ICE Candidate
浏览器 B
接收方
1 A 加入房间，获取本地媒体流
2 B 加入房间，A 收到通知
3 A 创建 Offer → 设置本地描述 → 发送给 B
4 B 收到 Offer → 设置远端描述 → 创建 Answer → 发送给 A
5 双方交换 ICE Candidate
6 ICE 连通性检查完成 → P2P 连接建立
7 音视频数据 直接 P2P 传输，不再经过信令服务器
Signaling

信令服务器实现

Node.js + Socket.IO —— 只做转发，不处理媒体数据

import { Server } from 'socket.io'

const io = new Server(3000, { cors: { origin: '*' } })

io.on('connection', (socket) => {
  // 加入房间
  socket.on('join-room', (roomId) => {
    socket.join(roomId)
    socket.to(roomId).emit('user-joined', socket.id)
  })

  // 转发 SDP Offer / Answer
  socket.on('offer', ({ roomId, offer }) => {
    socket.to(roomId).emit('offer', { offer, from: socket.id })
  })
  socket.on('answer', ({ roomId, answer }) => {
    socket.to(roomId).emit('answer', { answer, from: socket.id })
  })

  // 转发 ICE Candidate
  socket.on('ice-candidate', ({ roomId, candidate }) => {
    socket.to(roomId).emit('ice-candidate', { candidate, from: socket.id })
  })
})

信令服务器 = "婚介所" —— 帮双方交换联系方式，之后它们直接对话

NAT Traversal

NAT 穿透原理

大多数设备在内网，没有独立公网 IP，需要"搭桥"才能找到对方

📞

STUN

"查号台"

帮助内网设备发现自己的公网 IP 和端口

📬

TURN

"传纸条的中间人"

P2P 失败时作为中继服务器转发数据

🧭

"智能导航"

// ==================== 加入房间 & 初始化 ====================
async function joinRoom() {
  // 1. 获取本地媒体流
  localStream = await navigator.mediaDevices.getUserMedia({
    video: { width: 1280, height: 720 },
    audio: true
  })
  localVideoRef.value!.srcObject = localStream

  // 2. 连接信令服务器
  socket = io('http://localhost:3000')
  socket.on('connect', () => {
    socket!.emit('join-room', { roomId: roomId.value }, (result) => {
      if (result.success) callState.value.isInRoom = true
    })
  })
  setupSocketListeners()
}

// ==================== Socket 事件监听 ====================
function setupSocketListeners() {
  // 新用户加入 → 创建 Offer
  socket!.on('user-joined', async (data) => {
    callState.value.remotePeerId = data.socketId
    await createPeerConnection()
    await createAndSendOffer(data.socketId)
  })

  // 收到 Offer → 创建 Answer
  socket!.on('offer', async (data) => {
    await createPeerConnection()
    await handleOffer(data)
  })

  // 收到 Answer
  socket!.on('answer', async (data) => {
    await handleAnswer(data)
  })

  // 收到 ICE Candidate
  socket!.on('ice-candidate', async (data) => {
    await handleIceCandidate(data)
  })
}
Practice

核心代码：PeerConnection 管理

async function createPeerConnection() {
  pc = new RTCPeerConnection({ iceServers: [
    { urls: 'stun:stun.l.google.com:19302' }
  ]})

  // 添加本地轨道
  localStream?.getTracks().forEach(track => {
    pc!.addTrack(track, localStream!)
  })

  // 监听远端轨道
  pc.ontrack = (event) => {
    remoteVideoRef.value!.srcObject = event.streams[0]
  }

  // 收集 ICE Candidate 并发送
  pc.onicecandidate = (event) => {
    if (event.candidate && callState.value.remotePeerId) {
      socket?.emit('ice-candidate', {
        targetId: callState.value.remotePeerId,
        candidate: event.candidate.toJSON()
      })
    }
  }

  // 连接状态变化
  pc.onconnectionstatechange = () => {
    callState.value.isConnected = (pc?.connectionState === 'connected')
  }
}

// 创建并发送 Offer（呼叫方）
async function createAndSendOffer(targetId: string) {
  dataChannel = pc!.createDataChannel('chat', { ordered: true })
  const offer = await pc!.createOffer()
  await pc!.setLocalDescription(offer)
  socket?.emit('offer', { targetId, sdp: offer })
}

Comparison

WebRTC vs WebSocket vs WebTransport

维度 WebRTC WebSocket WebTransport
传输层 UDP（可靠/不可靠可选） TCP QUIC (基于 UDP)
连接方式 P2P C/S C/S
延迟 极低 (20~80ms) 中等 (50~200ms) 低 (30~100ms)
带宽成本 低（P2P） 高（服务器中转） 高（服务器中转）
适用场景 音视频、实时游戏、文件传输 文字聊天、信令、通知 游戏、实时流
连接复杂度 高（SDP + ICE + 信令） 低（一次握手） 中（QUIC 握手）
加密 强制 DTLS/SRTP 可选 TLS (wss://) 强制 TLS 1.3

选择建议： 需要音视频 → WebRTC | 文字/信令 → WebSocket | 新一代低延迟 → WebTransport

Debug

常见问题 & 调试技巧

🔍 chrome://webrtc-internals

  • 查看所有活跃 RTCPeerConnection
  • 检查 signalingState / connectionState
  • 查看 ICE 候选类型（host/srflx/relay）
  • 分析带宽、帧率、丢包率统计
  • 查看完整 SDP Offer/Answer 内容

🐛 常见问题

  • ICE failed → 添加 TURN 服务器
  • Permission denied → 检查 HTTPS / 权限
  • 一方看不到对方 → 确认 addTrack 在 createOffer 之前
  • Safari 兼容 → 用户交互后调用 getUserMedia
  • 移动端断开 → 处理前后台切换
